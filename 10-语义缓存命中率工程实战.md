# 语义缓存命中率从 12% 到 38%：4 个工程坑

OpenRouter 仪表盘上 2026 年 6 月 8-14 日那一周的数据放在我们运维群里，没人说话。**全球大模型周调用量 44.6 万亿 Token，环比 +23.5%，连续八周上涨**；其中**中国区 18.42 万亿，连续七周超过美国**。DeepSeek-V4-Flash 蝉联调用量榜首，已经第五周第一了。

调用量的第二张图永远是账单。我翻了一下我们自己上一财季的支出曲线：3 月 28 万、4 月 41 万、5 月 65 万、6 月还没结束已经冲到 78 万——和行业的 8 周连涨节奏几乎贴合。这个时候，不管路由表怎么调、再怎么把流量往便宜模型上压，**只要"调用次数"这条曲线还在涨，单价省下来的钱就总是在被调用次数吃回去**。

上一篇《分级路由策略实战》写完，我在评论区看到一句话：

> "路由表的天花板就是用户问什么模型答什么，只要用户在问，账单就在涨。"

这句话很扎心，但不准确。**用户在问，并不等于模型必须答**——如果他问的问题，过去一周里已经有人问过，且我们存了答案，理论上就可以**根本不调用 LLM**。这就是语义缓存（Semantic Cache）的全部价值：**省的不是单价，是调用次数本身**。

但我必须坦白：我们一开始的语义缓存命中率只有 **12%**，和行业普遍水平差不多。这个数字意味着 100 次请求里只有 12 次能跳过 LLM，剩下 88 次老老实实掏钱。我那时候一度觉得"语义缓存是个噱头"——直到接下来半年，我们把命中率从 12% 一路拧到 **38%**，月度模型支出从 $45K 砍到 $14K，我才意识到，**问题不在语义缓存这个概念上，问题在工程实现上**。

更让人不舒服的事实是：我自己在外面分享、看到的二十几家做 LLM 中间件的团队，**80% 的命中率都没有突破 15%**。这不是技术不行，是大家都在同一组坑里挣扎。这一篇就把那 4 个坑掰开揉碎讲清楚，每一个坑给出代码、给出数据、给出我们当时是怎么爬出来的。文末有 4 周落地路线，照着走。

## 一、为什么 12% 是行业基线

每次有人来问"语义缓存到底能命中多少"，我都会先反问一句：你说的"语义缓存"指的是哪一类？因为这词被滥用得很厉害。市面上至少有三种都被叫做"缓存"的东西，命中率上限完全不同：

| 类型 | 实现方式 | 命中率上限 | 备注 |
|------|---------|-----------|------|
| 精确缓存 | hash(query) 做 key | < 10% | 用户措辞稍变就 miss |
| 语义缓存（粗调） | embedding + 固定阈值 | 12-15% | 大多数团队的起点 |
| 语义缓存（精调） | 分层 Key + 动态阈值 + 失效策略 | 30-45% | 需要工程化投入 |
| Prompt Cache（厂商侧） | LLM 厂商自带 | 0%（不算命中只算折扣） | 是"输入价折扣"不是"调用次数减少" |

第四种 Prompt Cache 这两年 Anthropic 和 DeepSeek 都在主推，但严格说它**根本不是缓存**——它只是把同样的输入算便宜一点，请求依然要发出去、依然要算 output token、依然占用速率配额。我在第 2 篇《Token 成本优化实战》里专门拆过它的账，本篇不再展开。

回到主线，**精确缓存为什么命中率上不去**？我们在生产数据里抽过一组样本：8000 条客服 query，用户问的其实只有 217 个"语义意图"，但这 8000 条 query 写成了 6342 种不同的字面表达。换句话说，**从字面到字面的去重率只有 21%**。再考虑空格、标点、emoji、错别字、多余的礼貌用语，最后能 hash 到同一个 key 的不到 8%。

| 同一意图的字面变体 | 出现次数 |
|------------------|---------|
| "怎么退货？" | 412 |
| "请问退货流程是什么呀" | 287 |
| "我要退货 怎么办" | 213 |
| "想退货，你们这边走什么手续" | 156 |
| "退货" | 128 |
| "退货怎么操作" | 97 |
| "刚收到的东西想退掉" | 64 |
| 其他 56 种变体 | < 50 / 种 |

精确缓存能做的事，仅仅是把 412 次变成 411 次缓存命中——后面那 50+ 种变体它一个都接不住。

那为什么"语义缓存粗调"也只能到 12-15%？这就要说到第一个坑，但在那之前，我先把全文的路线图摆出来，你心里有数后续在讲什么。

## 二、全文路线图

下面这张表是我们半年的命中率走势，每一行对应一个坑修完之后的稳态值。这也是后续 4 个章节的展开顺序：

| 阶段 | 坑名 | 命中率 | 工程难度 | 关键产出 |
|------|------|--------|---------|---------|
| 起点 | naive 语义缓存 | 12% | 低 | 跑通即可 |
| 坑 1 | Embedding 选型与阈值校准 | 12% → 22% | 中 | 阈值 ROC 曲线、模型对比表 |
| 坑 2 | 缓存 Key 设计 | 22% → 30% | 高 | 分层 Key、租户隔离方案 |
| 坑 3 | 失效策略与时效 | 30% → 35% | 中 | TTL 路由器、失效矩阵 |
| 坑 4 | 缓存击穿与预热 | 35% → 38% | 低（隐藏成本高） | singleflight、预热脚本 |

后面四个坑的章节都按相同结构组织：

1. **现象**：当时是怎么发现这是个坑的（往往是某次事故、某次成本暴涨）；
2. **根因**：为什么它会成为坑（背后的工程原理）；
3. **代码**：我们最终是怎么解的，给完整可跑的 demo；
4. **数据**：修完之后命中率 / 误命中率 / 延迟的真实变化。

跑题前再多说一句：**这 4 个坑的修复有强弱顺序**。坑 1（阈值）必须先修，否则坑 2、3、4 都建立在错误的命中判定上；坑 2、3、4 之间相对独立，你可以根据团队优先级调整。但跳过坑 1 直接做坑 2，结果就是命中率虚高、误命中率失控、用户开始投诉"答非所问"——这是我们见过最常见的反模式。

## 三、坑 1：Embedding 选型与阈值校准

### 3.1 现象：命中率上去了，但用户开始抱怨

我们第一版语义缓存上线时，命中率从 0 直接跳到 12%。第一周大家都很开心，第二周客服那边转来一个工单：

> 用户问"我的优惠券到期了能延期吗"，系统直接回了"你的会员有效期还剩 3 个月"——驴唇不对马嘴。

我们查了下日志，这条 query 的 embedding 和缓存里"我的会员还有多久过期"的余弦相似度是 0.84，刚好压过我们当时设的阈值 0.80。从 embedding 模型的角度看，这两句话确实都涉及"我的XX 还有多久"，属于同一个语义簇。但从业务角度看，**"优惠券"和"会员"是两个完全不同的实体**，混了就是事故。

类似的工单一周收了 11 条，分别长得像这样：

- "苹果手机能下单吗" 命中了 "苹果电脑能下单吗" 的缓存（0.86）
- "我儿子的账号怎么开" 命中了 "我女儿的账号怎么开" 的缓存（0.91）
- "深圳能送货上门吗" 命中了 "上海能送货上门吗" 的缓存（0.88）

每一条都很可笑，但每一条都"语义上看起来很像"。我们意识到，**默认阈值是错的，模型选型也是错的**——但具体错在哪儿，得靠数据说话。

### 3.2 根因：Embedding 模型不是为缓存判定设计的

主流的 Embedding 模型（OpenAI 的 text-embedding-3-small、智源的 bge-large-zh、阿里的 gte-large 等）都是用对比学习训练的，目标是让"相关文本"在向量空间里聚得近一点。但"相关"和"等价"是两回事：

| 关系 | 余弦相似度区间（中文 BGE） | 适合做缓存命中吗 |
|------|--------------------------|----------------|
| 完全等价（仅措辞不同） | 0.92 - 1.00 | 可以 |
| 同主题不同问题 | 0.82 - 0.92 | **不可以** |
| 相关话题 | 0.70 - 0.82 | 不可以 |
| 弱相关 | 0.50 - 0.70 | 不可以 |

模型本身不会区分"等价"和"同主题"——区分它们的是**阈值**。阈值定低了，"同主题不同问题"被误判为命中；定高了，"等价"也被拒之门外。

更麻烦的是，**不同业务场景的等价线不在同一个数值上**。我们后来给三类业务做了独立的阈值校准，结论差异很大：

| 场景 | 推荐阈值 | 原因 |
|------|---------|------|
| 中文客服对话 | 0.86 | 实体词替换敏感（"优惠券" vs "会员"） |
| 英文技术问答 | 0.83 | 语法结构变化大、同义词更多 |
| 代码生成（多语言混合） | 0.89 | 微小符号差异往往代表完全不同的逻辑 |

那么阈值到底怎么定？拍脑袋不行，看模型论文也不行——必须用**自己的业务数据**跑 ROC 曲线。

### 3.3 代码：用 ROC 曲线找阈值的拐点

下面这段代码是我们生产里实际跑过的版本（去掉了脏数据清洗的部分），输入是一份人工标注的"等价对 / 非等价对"数据集，输出是一张阈值-命中率-误命中率三条曲线图，肉眼能直接看出拐点。

```python
"""
Embedding 阈值 ROC 校准脚本
输入：labeled_pairs.jsonl，每行 {"q1": str, "q2": str, "label": 1|0}
      label=1 表示业务上等价，label=0 表示不等价
输出：threshold_roc.png + 推荐阈值（最大化 F1 的点）
依赖：sentence-transformers, numpy, matplotlib, scikit-learn
"""
import json
import numpy as np
import matplotlib.pyplot as plt
from sentence_transformers import SentenceTransformer
from sklearn.metrics import roc_curve, f1_score


def load_pairs(path: str):
    pairs, labels = [], []
    with open(path, "r", encoding="utf-8") as f:
        for line in f:
            obj = json.loads(line)
            pairs.append((obj["q1"], obj["q2"]))
            labels.append(int(obj["label"]))
    return pairs, np.array(labels)


def compute_similarities(pairs, model_name: str):
    model = SentenceTransformer(model_name)
    q1_list = [p[0] for p in pairs]
    q2_list = [p[1] for p in pairs]
    v1 = model.encode(q1_list, normalize_embeddings=True, batch_size=64)
    v2 = model.encode(q2_list, normalize_embeddings=True, batch_size=64)
    # normalize 后点积 = 余弦相似度
    sims = np.sum(v1 * v2, axis=1)
    return sims


def sweep_thresholds(sims, labels, lo=0.70, hi=0.99, step=0.005):
    """扫描阈值，返回每个阈值下的 hit_rate / false_hit_rate / f1"""
    results = []
    for thr in np.arange(lo, hi, step):
        pred = (sims >= thr).astype(int)
        tp = int(((pred == 1) & (labels == 1)).sum())
        fp = int(((pred == 1) & (labels == 0)).sum())
        fn = int(((pred == 0) & (labels == 1)).sum())
        tn = int(((pred == 0) & (labels == 0)).sum())
        hit_rate = tp / max(tp + fn, 1)            # 业务等价被识别为命中的比例
        false_hit = fp / max(fp + tn, 1)           # 业务不等价被误判为命中的比例
        f1 = f1_score(labels, pred, zero_division=0)
        results.append((thr, hit_rate, false_hit, f1))
    return results


def plot_and_pick(results, model_name: str):
    thrs = [r[0] for r in results]
    hits = [r[1] for r in results]
    fhs = [r[2] for r in results]
    f1s = [r[3] for r in results]

    fig, ax1 = plt.subplots(figsize=(9, 5))
    ax1.plot(thrs, hits, label="hit_rate", color="#2E86AB")
    ax1.plot(thrs, fhs, label="false_hit_rate", color="#E63946")
    ax1.set_xlabel("threshold (cosine)")
    ax1.set_ylabel("rate")
    ax1.legend(loc="center left")

    ax2 = ax1.twinx()
    ax2.plot(thrs, f1s, label="f1", color="#06A77D", linestyle="--")
    ax2.set_ylabel("f1")
    ax2.legend(loc="center right")

    plt.title(f"Threshold ROC - {model_name}")
    plt.tight_layout()
    plt.savefig("threshold_roc.png", dpi=150)

    best_idx = int(np.argmax(f1s))
    best_thr, best_f1 = thrs[best_idx], f1s[best_idx]
    print(f"[{model_name}] best_threshold={best_thr:.3f}  f1={best_f1:.3f}  "
          f"hit={hits[best_idx]:.3f}  false_hit={fhs[best_idx]:.3f}")
    return best_thr


if __name__ == "__main__":
    pairs, labels = load_pairs("labeled_pairs.jsonl")
    print(f"loaded {len(pairs)} pairs, positive rate = {labels.mean():.3f}")

    # 三个候选模型同台比一下
    candidates = [
        "BAAI/bge-large-zh-v1.5",
        "BAAI/bge-m3",
        "intfloat/multilingual-e5-large",
    ]
    for m in candidates:
        sims = compute_similarities(pairs, m)
        results = sweep_thresholds(sims, labels)
        plot_and_pick(results, m)
```

跑完这段脚本你会得到一张图，长得大概像下面这样（用文字示意，实际是连续曲线）：

```
threshold  →  0.75   0.80   0.85   0.86   0.88   0.90   0.92
hit_rate    0.94   0.88   0.74   0.70   0.58   0.41   0.22
false_hit   0.31   0.18   0.07   0.04   0.02   0.01   0.00
f1          0.62   0.71   0.80   0.81   0.78   0.65   0.42
```

拐点很明显在 **0.86 附近**——再往下 false_hit 暴涨，再往上 hit_rate 暴跌。这个数字就是我们后面写在配置文件里的中文客服阈值。

### 3.4 三种 Embedding 模型的实测对比

跑同一份 8000 对标注数据（人工标注是哪些 query 业务上等价），三个候选模型在最优阈值下的对比：

| 模型 | 维度 | 最优阈值 | F1 | 推理延迟（CPU/单条） | 推理延迟（GPU/单条） |
|------|------|---------|-----|-------------------|-------------------|
| text-embedding-3-small（OpenAI） | 1536 | 0.84 | 0.79 | 网络往返 80-150ms | - |
| bge-large-zh-v1.5 | 1024 | 0.86 | 0.83 | 45ms | 8ms |
| bge-m3（多语言） | 1024 | 0.85 | 0.81 | 60ms | 11ms |
| 自训练（在 bge-base 上微调 5 万条） | 768 | 0.82 | 0.87 | 28ms | 5ms |

几个直接的结论，方便你跳过试错：

1. **OpenAI 的 text-embedding 模型不是最佳选择**——不仅 F1 不如 bge，更要命的是它要走 API，每次缓存查询都得多花 80-150ms 网络 RTT，缓存的延迟优势直接被吃掉一半。
2. **bge-large-zh-v1.5 是中文场景的默认起点**。3 行代码起步，F1 0.83 已经能把命中率从 12% 拉到 22% 左右。
3. **如果你有 5 万条以上的标注数据，自训练的收益最大**。我们后来在 bge-base-zh 上用 contrastive loss 微调了 4.7 万条业务对，F1 直接到 0.87，命中率又涨了 3 个点。但这步对没有 ML 团队的项目不友好，可以放到 v3。
4. **维度不是越高越好**。1536 vs 1024 在 F1 上没看出明显差异，但向量库的内存占用和检索延迟差 50%。

### 3.5 检索引擎：HNSW vs IVF vs 暴力

阈值定了、模型选了，下一个问题是怎么存这些向量。我们生产里的演进路径：

- **0-50K 条**：直接 numpy 数组在内存里点积。简单暴力，单次查询 1-3ms。
- **50K-500K 条**：Faiss 的 IndexFlatIP（暴力检索 GPU 加速）。延迟 0.5-2ms。
- **500K-10M 条**：Faiss 的 HNSW32（M=32, efConstruction=200）。延迟 1-5ms，内存占用是向量原始大小的 1.5 倍左右。
- **10M+ 条**：Milvus 或 Redis Stack 的 FT.SEARCH KNN。这一档基本就要走分布式向量库了，但缓存到这个体量也意味着你的业务足够大、值得专门维护一个团队。

我们当前生产规模大约 80 万条向量，用的是 Faiss HNSW 单机版本，部署在缓存服务的同进程里以避免跨进程序列化开销。一段最小化实现：

```python
import faiss
import numpy as np

class HNSWStore:
    def __init__(self, dim=1024, M=32, ef=128):
        self.index = faiss.IndexHNSWFlat(dim, M, faiss.METRIC_INNER_PRODUCT)
        self.index.hnsw.efConstruction = 200
        self.index.hnsw.efSearch = ef
        self.id_to_payload = {}     # 向量 id -> {"answer": str, "meta": dict}

    def add(self, vec: np.ndarray, payload: dict):
        idx = self.index.ntotal
        self.index.add(vec.reshape(1, -1).astype(np.float32))
        self.id_to_payload[idx] = payload
        return idx

    def search(self, vec: np.ndarray, k=5):
        D, I = self.index.search(vec.reshape(1, -1).astype(np.float32), k)
        return [(int(I[0][i]), float(D[0][i]), self.id_to_payload.get(int(I[0][i])))
                for i in range(k) if I[0][i] >= 0]
```

到这里，第一个坑就修完了：**模型从默认的 OpenAI 换到 bge-large-zh-v1.5**、**阈值从拍脑袋的 0.80 改成 ROC 找出来的 0.86**、**检索引擎从 numpy 升级到 Faiss HNSW**。命中率从 12% 到 22%，误命中率从 18% 降到 4% 以下，用户那边的"答非所问"工单从每周 11 条降到每周 0-1 条。

但 22% 离 38% 还远——下一个坑是 Key 怎么设计。

## 四、坑 2：缓存 Key 设计

### 4.1 现象：跨租户污染事故

修完阈值大概一个月后，我们出了一次更大的事故。一个 To B 客户在内部群里炸了：

> "我问'我们公司的退货政策是什么'，系统给我回了另一家公司的退货流程！这是数据泄漏！"

我后台一查冷汗就下来了。两家客户用的是同一个 SaaS 实例、同一个语义缓存。他们的 system prompt 不一样（各自注入了自己公司的政策、产品手册），但 user query 经过 embedding 后落在了同一个语义簇——余弦相似度 0.94，远高于阈值。**结果就是 A 公司的答案被 B 公司命中了**。

这不是命中率高低的问题，这是**合规事故**。当晚我们紧急上线了租户隔离补丁，第二天复盘的时候大家承认：**第一版的 Key 设计就是错的**。

错在哪儿？我们当时的逻辑很"朴素"：

```python
# 错误版本：只用 user_query 的 embedding 做 key
def cache_key_v1(user_query: str) -> bytes:
    return embedder.encode(user_query, normalize_embeddings=True)
```

这套逻辑隐含了三个假设，每一个都会出问题：

1. **假设所有 system prompt 一样** —— 多租户场景直接破功；
2. **假设所有 model 一样** —— 同样的 query，让 GPT-5 答和让 DeepSeek 答，风格、长度、格式可能完全不同，缓存混用会让用户看到风格突变；
3. **假设所有调用参数一样** —— 同样的 query 在 temperature=0 和 temperature=1 下输出差异巨大，前者适合做缓存源，后者根本不该缓存。

### 4.2 根因：缓存 Key 必须分层

把上面三个假设全部破开，缓存 Key 至少要承载这些信息：

```
租户 ID（tenant_id）
模型 ID（model）
prompt 模板版本（prompt_version）
调用参数（temperature, top_p, max_tokens 等关键参数）
user_query 的语义向量
```

但全部塞进一个 hash 是不行的——那样就退化成精确缓存了，命中率反而掉。正确的做法是 **L1（精确） + L2（语义）双层 Key**：

```
L1 (Exact, hash)
   → tenant_id + model + prompt_version + param_hash    （决定"这条缓存属于哪个上下文环境"）
   ↓ 命中后进入对应分区
L2 (Semantic, vector)
   → embedding(user_query)                              （决定"这条 query 在该环境下能否复用某条历史答案"）
```

把它画成一个稍微具象一点的图：

```
[Request]
    │
    ├─ tenant_id = "acme"
    ├─ model     = "deepseek-v4-flash"
    ├─ prompt_v  = "v3.1"
    ├─ params    = {temperature: 0, top_p: 1, max_tokens: 1024}
    └─ user_query= "我们公司的退货政策是什么"
            │
            ▼
    L1_key = sha256("acme|deepseek-v4-flash|v3.1|temp=0,top_p=1,max=1024")
            │
            ▼
    在 Redis 找 namespace = L1_key 下的向量分片 → 调 Faiss → embedding 命中判定
```

L1 命中失败时（即这个上下文环境是首次出现），L2 整个分片是空的，等于直接 miss 走 LLM 后回填新分片。L1 命中成功时，L2 才在该分片内做语义检索，**跨分片不可见**——这就是租户隔离的核心保证。

### 4.3 代码：双层 Key 的完整实现

下面是我们生产里真实在跑的简化版（去掉了监控埋点和异常处理，留下骨架）：

```python
"""
双层缓存 Key 实现：L1 精确分区 + L2 语义检索
存储：Redis（分区元数据 & 答案） + Faiss（每个分区一个 HNSW 索引）
"""
import hashlib
import json
import threading
from typing import Optional, Dict, Tuple
import numpy as np
import faiss
import redis
from sentence_transformers import SentenceTransformer

EMBED_DIM = 1024
DEFAULT_THRESHOLD = 0.86

embedder = SentenceTransformer("BAAI/bge-large-zh-v1.5")
rds = redis.Redis(host="localhost", port=6379)


def normalize_params(params: Dict) -> str:
    """只取影响输出的关键参数，按 key 排序后序列化"""
    keys_we_care = ("temperature", "top_p", "max_tokens",
                    "frequency_penalty", "presence_penalty")
    picked = {k: params.get(k) for k in keys_we_care if k in params}
    return json.dumps(picked, sort_keys=True)


def build_l1_key(tenant_id: str, model: str, prompt_v: str, params: Dict) -> str:
    raw = f"{tenant_id}|{model}|{prompt_v}|{normalize_params(params)}"
    return "scache:" + hashlib.sha256(raw.encode()).hexdigest()[:24]


class PartitionedSemanticCache:
    """每个 L1 分区一个独立的 Faiss 索引；分区索引按需懒加载，LRU 淘汰"""

    def __init__(self, max_partitions: int = 64):
        self._partitions: Dict[str, faiss.IndexHNSWFlat] = {}
        self._payloads: Dict[str, Dict[int, dict]] = {}
        self._lock = threading.Lock()
        self._max = max_partitions

    def _get_or_create(self, l1_key: str) -> Tuple[faiss.IndexHNSWFlat, Dict[int, dict]]:
        with self._lock:
            if l1_key in self._partitions:
                return self._partitions[l1_key], self._payloads[l1_key]
            if len(self._partitions) >= self._max:
                # 简化版 LRU：弹出第一个
                victim = next(iter(self._partitions))
                self._partitions.pop(victim)
                self._payloads.pop(victim)
            idx = faiss.IndexHNSWFlat(EMBED_DIM, 32, faiss.METRIC_INNER_PRODUCT)
            idx.hnsw.efConstruction = 200
            idx.hnsw.efSearch = 64
            self._partitions[l1_key] = idx
            self._payloads[l1_key] = {}
            # 从 Redis 把已有数据加载进来（生产里这步异步做）
            self._load_from_redis(l1_key, idx, self._payloads[l1_key])
            return idx, self._payloads[l1_key]

    def _load_from_redis(self, l1_key, idx, payloads):
        cnt = 0
        for raw in rds.lrange(f"{l1_key}:vecs", 0, -1):
            obj = json.loads(raw)
            v = np.array(obj["v"], dtype=np.float32).reshape(1, -1)
            idx.add(v)
            payloads[cnt] = {"q": obj["q"], "a": obj["a"], "ts": obj["ts"]}
            cnt += 1

    def lookup(self, tenant_id: str, model: str, prompt_v: str,
               params: Dict, user_query: str,
               threshold: float = DEFAULT_THRESHOLD) -> Optional[dict]:
        l1 = build_l1_key(tenant_id, model, prompt_v, params)
        idx, payloads = self._get_or_create(l1)
        if idx.ntotal == 0:
            return None
        vec = embedder.encode(user_query, normalize_embeddings=True).astype(np.float32)
        D, I = idx.search(vec.reshape(1, -1), 1)
        score, top = float(D[0][0]), int(I[0][0])
        if score >= threshold and top >= 0:
            hit = payloads[top]
            return {"answer": hit["a"], "score": score,
                    "matched_query": hit["q"], "ts": hit["ts"]}
        return None

    def insert(self, tenant_id: str, model: str, prompt_v: str,
               params: Dict, user_query: str, answer: str, ts: int):
        l1 = build_l1_key(tenant_id, model, prompt_v, params)
        idx, payloads = self._get_or_create(l1)
        vec = embedder.encode(user_query, normalize_embeddings=True).astype(np.float32)
        new_id = idx.ntotal
        idx.add(vec.reshape(1, -1))
        payloads[new_id] = {"q": user_query, "a": answer, "ts": ts}
        rds.rpush(f"{l1}:vecs", json.dumps({
            "v": vec.tolist(), "q": user_query, "a": answer, "ts": ts,
        }))
```

调用侧的样子：

```python
cache = PartitionedSemanticCache(max_partitions=128)

def chat(tenant_id, user_query, **kwargs):
    model = kwargs.get("model", "deepseek-v4-flash")
    prompt_v = kwargs.get("prompt_version", "v3.1")
    params = {"temperature": 0, "top_p": 1, "max_tokens": 1024}

    hit = cache.lookup(tenant_id, model, prompt_v, params, user_query)
    if hit:
        return {"text": hit["answer"], "from_cache": True,
                "matched": hit["matched_query"], "score": hit["score"]}

    answer = call_llm(model, system_prompt_for(tenant_id, prompt_v),
                      user_query, **params)
    cache.insert(tenant_id, model, prompt_v, params, user_query,
                 answer, ts=int(time.time()))
    return {"text": answer, "from_cache": False}
```

### 4.4 多租户的"隔离 + 共享"权衡

上面这个版本是**完全隔离**——每个租户一个分区，互不可见。这能避免数据泄漏，但有个副作用：**新租户的命中率从 0 开始**，需要积累一段时间才有缓存可命中。如果你的租户数有上百个，整体平均命中率会被新租户拖累。

我们后来又加了一个开关：**跨租户语义共享层**。原理是把"通用知识类"问题（"什么是 SaaS 订阅模式""怎么注册账号"这类不涉及租户私有信息的）抽出来放到一个共享分区：

| 问题类型 | 分区策略 | 例子 |
|---------|---------|------|
| 租户私有信息 | 严格隔离 | "我们公司的退货政策""我们的合同模板" |
| 通用产品知识 | 共享 | "怎么注册""怎么充值""支持哪些支付方式" |
| 行业通识 | 共享 | "什么是 ESG""GAAP 和 IFRS 区别" |

判定一条 query 是否可以走共享层，我们用了一个简单粗暴的方法：**在 system prompt 里 grep 一下私有标识符（公司名、产品名、合同号等）**，如果 user_query 没有引用任何私有标识符，就允许它进共享分区。这条规则误判率不到 0.5%，但成本上几乎没有任何增量。共享层上线后，新租户的冷启动命中率从 0% 提升到 14% 左右。

### 4.5 命中率的进一步爬升

把整个 Key 体系从 v1 升到 v2 之后，半个月的数据：

| 指标 | v1（单层 Key） | v2（双层 Key） |
|------|--------------|---------------|
| 命中率 | 22% | 30% |
| 误命中率 | 4.0% | 0.3% |
| 跨租户事故 | 1 起/月 | 0 |
| 单次查询延迟（P50） | 4ms | 6ms |
| 单次查询延迟（P99） | 12ms | 18ms |

代价是 P99 延迟多了 6ms（多了一次 hash + 分区索引切换），但这个量级用户完全感知不到。命中率提升 8 个百分点是更值钱的部分。

到这里命中率到 30%，下一个坑是 TTL。

## 五、坑 3：失效策略与时效

### 5.1 现象：缓存了"今天的订单状态"

修完 Key 之后我们以为可以坐等命中率慢慢爬，结果一个月后又被一个工单打脸。

> 用户问"我今天那个订单到哪儿了"，系统回答"已签收"——但用户根本没收到货。

查日志发现，两小时前另一个用户问过一模一样的句子，那时候订单确实已签收。我们的缓存 TTL 默认设了 24 小时，于是这个"已签收"的回答被复用了。这一次没有跨租户问题、没有 embedding 问题、没有 Key 问题——是**纯粹的时效问题**：**有些信息根本不该被缓存这么久**。

### 5.2 根因：内容时效有三档，TTL 必须分级

我们事后做了一次内容时效梳理，把所有可能进缓存的 query 大致分成三类：

| 类别 | 时效 | 例子 | 推荐 TTL |
|------|------|------|---------|
| 通用知识 | 月-年 | "什么是 ESG""退货流程是什么" | 30 天 |
| 业务知识 | 周-天 | "本月活动有哪些""新版本功能" | 24 小时（且 T+1 强制刷新） |
| 实时数据 | 分钟-秒 | "我的订单状态""今天天气" | **禁用缓存** |

第一类东西放半年都没事。第二类放 24 小时也行，但要做 T+1 校验——我们现在每天凌晨 3 点跑一个 job，把过去一天命中过的"业务知识类"缓存条目主动失效，等下次访问时再重建。第三类就是这次出事的，**根本不该进缓存**。

那怎么自动判断一条 query 属于哪一类？我们走过两个路径：

- **v1：关键词规则** —— 维护一个停用关键词表（"今天""现在""我的订单""我的账户""刚刚""最新"等），命中即禁用缓存。简单粗暴，但漏判率高（用户问"刚下的单到哪了"，"刚"匹配不上"刚刚"）。
- **v2：分类器 + 关键词** —— 用一个轻量级的意图分类器（蒸馏自 LLM 的小模型，约 50MB）先粗分，再叠加关键词兜底。漏判率从 v1 的 8% 降到 1.2%。

下面是 v2 的实现骨架。

### 5.3 代码：分级 TTL 路由器

```python
"""
分级 TTL 路由器：根据 query 的"时效类别"自动选择缓存策略
- TIMELESS: 通用知识，TTL 30 天
- WEEKLY:   业务知识，TTL 24 小时 + T+1 主动刷新
- REALTIME: 实时数据，禁用缓存
"""
import re
from dataclasses import dataclass
from typing import Literal, Optional

# 实时关键词：命中任意一个即视为 REALTIME
REALTIME_PATTERNS = [
    r"今天|今日|今晚|今早|当前|目前|现在|刚才|刚刚|刚下|刚买",
    r"我的订单|我的账户|我的余额|我的卡|我的会员",
    r"最新|实时|此刻|这会儿",
    r"\d{4}年\d{1,2}月\d{1,2}日",      # 具体日期
    r"还剩|还有多少|进度",
]
_REALTIME_RE = re.compile("|".join(f"({p})" for p in REALTIME_PATTERNS))

# 业务知识关键词（带"本/这"等指代当前时间段的）
WEEKLY_PATTERNS = [
    r"本月|本周|这个月|这周|这次活动|当前活动",
    r"新版本|新功能|最新发布",
]
_WEEKLY_RE = re.compile("|".join(f"({p})" for p in WEEKLY_PATTERNS))


@dataclass
class CachePolicy:
    category: Literal["TIMELESS", "WEEKLY", "REALTIME"]
    ttl_seconds: Optional[int]    # None 表示禁用缓存
    refresh_at_dawn: bool         # 是否凌晨主动刷新


def classify_query(query: str) -> CachePolicy:
    if _REALTIME_RE.search(query):
        return CachePolicy("REALTIME", None, False)
    if _WEEKLY_RE.search(query):
        return CachePolicy("WEEKLY", 24 * 3600, True)
    # 默认归到 TIMELESS（生产里这里会先调一次轻量级分类器再 fallback）
    return CachePolicy("TIMELESS", 30 * 86400, False)


# 在 cache.lookup / cache.insert 之前调用
def should_use_cache(query: str) -> bool:
    return classify_query(query).category != "REALTIME"


def cache_ttl_for(query: str) -> Optional[int]:
    return classify_query(query).ttl_seconds
```

### 5.4 失效矩阵：把规则画在墙上

代码可以解决"自动分类"这一步，但策略本身的制定需要团队讨论。我们后来在墙上贴了一张矩阵图，团队里产品、研发、客服三方都来填——内容时效是产品最懂的，业务可容忍延迟是客服最懂的，技术实现是研发最懂的。

| 业务问题 | 内容时效 | 业务可容忍延迟 | 决策 |
|---------|---------|--------------|------|
| 退货政策 | 季度更新 | 答错 = 客诉 | 缓存 30 天 + 政策变更时手动 invalidate |
| 本月活动 | 周更新 | 答错 = 漏单 | 缓存 24h + 凌晨刷新 |
| 商品价格 | 实时 | 答错 = 价格欺诈 | **禁用缓存** |
| 订单状态 | 实时 | 答错 = 重大客诉 | **禁用缓存** |
| 账户余额 | 实时 | 涉及资金 | **禁用缓存** |
| 会员等级规则 | 半年更新 | 答错 = 客诉 | 缓存 30 天 |
| 客服营业时间 | 季度更新 | 容忍度高 | 缓存 30 天 |
| 物流时效（"广州到北京几天"） | 月度更新 | 容忍度高 | 缓存 7 天 |

这张表的好处是它把决策权下沉到每个具体业务上——而不是工程师一个人在配置文件里拍 TTL。我们后来把它存成 YAML，热更新生效。

## 六、坑 4：缓存击穿与预热

### 6.1 现象：双十一前夜的雪崩

去年双十一前一天晚上 8 点，我们做了一次大促预热：客服系统挂了一条公告说"双十一活动规则已更新，请咨询客服"。结果接下来的 5 分钟里：

- "双十一有什么活动" 这条 query 被并发问了 1247 次
- 我们的语义缓存里**没有这条**——因为活动规则刚更新，旧缓存被 invalidate 了
- 1247 个请求**全部 miss**、**全部并发打到 LLM**
- LLM 厂商的速率限制被瞬时打爆，返回 429
- 速率限制触发后我们的降级链路把流量切到备用 provider，备用 provider 也被打爆
- 最终用户看到的是"客服繁忙，请稍后再试"

这就是经典的**缓存击穿**——同一个 hot query 同时 miss，所有请求都去查后端，等于缓存层完全失效。

### 6.2 根因：miss 必须收敛，TTL 必须打散

缓存击穿和缓存雪崩是两个相关但不同的问题：

| 问题 | 触发条件 | 后果 |
|------|---------|------|
| 缓存穿透 | query 永远没有缓存（恶意构造、长尾） | 后端持续被打 |
| 缓存击穿 | 某个 hot key 缓存失效瞬间，并发 miss | 后端瞬时被 N 倍流量打 |
| 缓存雪崩 | 大量缓存同时过期 | 后端持续被打 |

我们这次出的事是击穿。解决思路有三条，都得做：

1. **请求收敛（singleflight）**：1247 个并发 miss，让其中 1 个去打 LLM，其他 1246 个等结果。
2. **TTL 抖动**：所有缓存的实际 TTL 在基础值 ±20% 之间随机，避免集中失效。
3. **预热（warmup）**：上线大版本 / 大促前，把已知的高频 query 提前生成好缓存。

### 6.3 代码：singleflight 的 Python 实现

Go 标准库里有 `golang.org/x/sync/singleflight`，是这个模式的经典实现。Python 没有官方版本，下面这段是我们自己写的（无外部依赖，asyncio 友好）：

```python
"""
asyncio 版 singleflight：相同 key 的并发调用只发起一次实际请求
"""
import asyncio
from typing import Any, Awaitable, Callable, Dict


class SingleFlight:
    def __init__(self):
        self._inflight: Dict[str, asyncio.Future] = {}
        self._lock = asyncio.Lock()

    async def do(self, key: str, fn: Callable[[], Awaitable[Any]]) -> Any:
        async with self._lock:
            if key in self._inflight:
                fut = self._inflight[key]
                # 已经有人在跑，等结果
                async with self._unlocked():
                    return await fut
            fut = asyncio.get_event_loop().create_future()
            self._inflight[key] = fut

        try:
            result = await fn()
            fut.set_result(result)
            return result
        except Exception as e:
            fut.set_exception(e)
            raise
        finally:
            async with self._lock:
                self._inflight.pop(key, None)

    class _unlocked:
        # 简化的"反锁"上下文，实际生产里用更稳的实现
        def __init__(self): ...
        async def __aenter__(self): pass
        async def __aexit__(self, *a): pass


# ============ 集成到缓存调用链 ============
sf = SingleFlight()

async def chat_with_singleflight(tenant_id, user_query, **kwargs):
    policy = classify_query(user_query)

    if policy.category == "REALTIME":
        return await call_llm_async(tenant_id, user_query, **kwargs)

    # 先查缓存
    hit = cache.lookup(tenant_id, ..., user_query)
    if hit:
        return {"text": hit["answer"], "from_cache": True}

    # miss 时用 singleflight 收敛
    sf_key = f"{tenant_id}:{hashlib.md5(user_query.encode()).hexdigest()[:16]}"

    async def _miss_handler():
        # 双重检查：可能在等锁的过程中已被另一个请求填充
        hit2 = cache.lookup(tenant_id, ..., user_query)
        if hit2:
            return hit2["answer"]
        answer = await call_llm_async(tenant_id, user_query, **kwargs)
        cache.insert(tenant_id, ..., user_query, answer,
                     ts=int(time.time()), ttl=jittered_ttl(policy.ttl_seconds))
        return answer

    answer = await sf.do(sf_key, _miss_handler)
    return {"text": answer, "from_cache": False}


def jittered_ttl(base: int) -> int:
    """基础 TTL ±20% 抖动，避免雪崩"""
    import random
    if base is None:
        return None
    return int(base * (0.8 + random.random() * 0.4))
```

注意几个细节：

1. **singleflight 的 key 不能用 embedding**——embedding 不稳定（同样的 query 可能因模型版本细微差异得到几乎相同但不完全相同的向量），用 user_query 的 hash 即可。
2. **双重检查（double-check）必须有**——一个并发组里第一个完成的请求会把答案写回缓存，但 singleflight 在 Python asyncio 下不保证后续请求一定能看到这个写入，所以进入 miss handler 的第一步就要再 lookup 一次。
3. **超时控制**：singleflight 等待的请求要有 timeout，否则一个慢请求会拖死整组。我们生产里给的是 8 秒，比单次 LLM 调用的最大超时多 1 秒。

### 6.4 预热脚本：用历史 query 把缓存焐热

singleflight 解决"miss 风暴"的瞬时问题，但解决不了"缓存里压根没有这条 hot query"的根本问题。预热是补这个空缺的：

```python
"""
缓存预热脚本：拉过去 N 天的 top K query，逐条生成缓存
建议在上线大版本 / 大促开始前 24h 跑
"""
import time
import asyncio
from collections import Counter

async def warmup_topk(days: int = 30, top_k: int = 5000,
                     concurrency: int = 8):
    # 1. 从日志聚合 top query（生产里走 ClickHouse / Hive）
    raw = fetch_query_logs(days=days)
    counter = Counter(canonicalize(q) for q in raw)
    hot_queries = [q for q, _ in counter.most_common(top_k)]
    print(f"warmup planning: {len(hot_queries)} unique queries")

    # 2. 过滤掉 REALTIME 类（不该预热）
    hot_queries = [q for q in hot_queries
                   if classify_query(q).category != "REALTIME"]

    # 3. 限流并发拉答案 + 写缓存
    sem = asyncio.Semaphore(concurrency)
    success, failed = 0, 0

    async def warm_one(q: str):
        nonlocal success, failed
        async with sem:
            try:
                # 用默认租户 + 公开 prompt 模板
                if cache.lookup(DEFAULT_TENANT, ..., q):
                    return
                answer = await call_llm_async(DEFAULT_TENANT, q,
                                              model="deepseek-v4-flash")
                cache.insert(DEFAULT_TENANT, ..., q, answer,
                             ts=int(time.time()),
                             ttl=jittered_ttl(7 * 86400))
                success += 1
            except Exception as e:
                failed += 1

    await asyncio.gather(*(warm_one(q) for q in hot_queries))
    print(f"warmup done: success={success}, failed={failed}")


def canonicalize(q: str) -> str:
    """简单规范化：去多余空格、统一标点、小写化英文"""
    import re
    q = q.strip()
    q = re.sub(r"\s+", " ", q)
    q = q.replace("？", "?").replace("！", "!")
    return q
```

预热的两个隐藏成本必须提一下：

1. **预热本身要花钱**。5000 条 query × 平均 600 token 输出 × 0.5 元/百万 token ≈ 1.5 元一次预热，看起来便宜。但如果你的命中率分母（请求量）不够大，预热的钱可能比缓存省下来的钱还多——所以预热只对**月调用量 50 万次以上**的系统有正收益。
2. **预热的内容会随时间老化**。我们一开始把预热设成"上线时跑一次"，结果一个月后预热的那批答案有 12% 已经因为业务变化过期了，反而成了脏数据源。后来改成**每周一凌晨 3 点跑一次** + **大促前手动跑一次**。

### 6.5 数据：最后 3 个百分点的真实代价

加完 singleflight + TTL 抖动 + 预热之后：

| 指标 | 加击穿防御前 | 加击穿防御后 |
|------|-----------|-----------|
| 命中率 | 35% | 38% |
| 命中率峰谷波动 | ±9pp | ±2pp |
| LLM 厂商 429 错误 | 14 次/周 | 0-1 次/周 |
| P99 延迟（命中路径） | 22ms | 24ms |
| P99 延迟（miss 路径） | 1.8s | 1.8s（无变化） |

命中率提升只有 3 个点，但**命中率波动从 ±9pp 收敛到 ±2pp**——这才是这个坑真正的价值。之前业务侧总抱怨"今天命中率怎么又跌到 25% 了"，现在曲线基本贴着 38% 这条线走，业务方对成本的预测准确度高了一截。

至此 4 个坑全部修完，命中率 38%。下一节把整个半年的曲线连起来回顾。

## 七、从 12% 到 38% 的实测路径

### 7.1 半年命中率走势

下面这张表是脱敏后我们自己生产的真实数据，月度均值。月调用量这一栏没有完全展示绝对值，做了归一化处理（M1 起点 = 1.0），但比例是真实的：

| 月份 | 阶段 | 命中率（月均） | 单月调用量（归一化）| 月度模型支出 | 单位调用成本（相对值） |
|------|------|--------------|------------------|------------|--------------------|
| M1 | naive 起点 | 12% | 1.00 | $45.2K | 1.00 |
| M2 | 修阈值（坑1）| 22% | 1.08 | $39.8K | 0.81 |
| M3 | 修阈值稳定 | 23% | 1.15 | $40.6K | 0.78 |
| M4 | 修 Key（坑2）| 30% | 1.21 | $32.4K | 0.59 |

### 7.4 写给 ROI 计算者的备忘

这半年我们投入的人力：

- 坑 1 阈值校准：1 个工程师 × 2 周（含数据标注 1 周）
- 坑 2 Key 改造：2 个工程师 × 4 周（含联调和回归测试）
- 坑 3 分级 TTL：1 个工程师 × 2 周 + 产品 0.5 周
- 坑 4 击穿防御：1 个工程师 × 1 周

总计约 **2 人 × 2 月**的人力投入，换来月度账单从 $45K 降到 $14K——按一年算就是 $370K 的支出节省。这个 ROI 高到让我每次回想都觉得不真实。

如果你的月调用量没那么大（比如月 5 万次以下），坦白讲做这套不划算——人力成本会比账单节省更高。**月调用量 50 万次以上是这套工程化投入的盈亏平衡线**——这也是开篇为什么把目标读者画到这条线上。

## 八、命中率天花板与下一步

### 8.1 38% 不是终点，但 40% 之后边际收益骤降

我们在 38% 这条线上稳了大约两个月，期间继续做小调优——把阈值在不同子场景下做更细的拆分（"售后类" 0.85，"咨询类" 0.87，"投诉类" 0.92），命中率慢慢爬到 39.6%。再往上动不了了。

为什么？因为**剩下那 60% 的 miss，其实是真的没有重复 query**。我们抽了一周的 miss 样本去做聚类，结果发现：

| miss 类型 | 占比 | 能否被语义缓存救回 |
|---------|------|-----------------|
| 长尾独特问题（一周内只出现 1-2 次） | 41% | 不能 |
| 含个人信息的 query（"我手机号 138xxxxxxxx 注册不上"）| 18% | 不能（且不该缓存） |
| 多轮对话上下文相关（依赖前文）| 15% | 当前架构不能，见 8.2 |
| 同义改写未识别 | 11% | **能，但需要新机制** |
| 实时数据 query | 8% | 不该缓存 |
| 噪声 / 测试流量 | 7% | 不该计入 |

也就是说，命中率的理论天花板大约在 60-65% 左右（前两类完全打不动），而我们离这个上限还有 20+ 个百分点的空间。但这 20 个百分点的难度**指数级上升**——继续在阈值和 Key 上抠没用了，需要做架构层面的事情。

### 8.2 想突破 50%+ 需要做的三件事

这一节属于"还在路上"的内容，我们当前正在做或者计划做，但还没有完整的生产数据。点到为止，下一篇会展开。

**1. 问题归一化层**

现状：用户问"怎么退货"和"退货怎么操作"会被 embedding 模型识别为很接近（相似度 0.91），命中。但"退货退款流程"、"退退退到底怎么退"这种就开始飘了。

我们正在试一个 **LLM-as-canonicalizer** 的方案：在 embedding 之前，先用一个轻量级模型（DeepSeek-V4-Flash 也够用）把 user_query 改写为一个"标准问法"，再去匹配缓存。粗测下来命中率能再涨 5-7 个百分点，但每条 query 多了一次 50ms 的 LLM 调用——**得失之间需要算账**。

**2. 多轮对话上下文压缩**

现状：客服系统里 60% 的对话超过 3 轮，但语义缓存只看当前这轮的 user_query。"它的价格是多少"这种回指（"它"指上文提过的某个商品）根本无法匹配缓存。

正在做的方案是 **rolling summary cache**：每轮对话结束后用一个小模型生成一个"上下文摘要"（120 token 以内），下一轮的语义缓存 key 用 `embedding(summary + new_query)` 而不是 `embedding(new_query)`。这个方案理论上能把多轮对话的命中率从 5% 拉到 25%+。但工程复杂度高，目前还在小流量灰度。

**3. 用户意图聚类的"模板缓存"**

现状：每条 query 在缓存里都是独立条目。但很多 query 其实是**模板的实例化**，比如"我的订单 12345 到哪了"和"我的订单 67890 到哪了"，模板都是"订单状态查询"，只是订单号不同。

正在做的方案是 **slot-based 模板缓存**：先做意图识别 + 槽位提取，把 query 解析成 `(intent: 订单状态查询, slot: {order_id: 12345})`，缓存的 key 是 `intent + frame template`，命中后再用 slot 实时替换答案里的具体值。这个方案对客服场景特别有效，预估能把"准实时数据"那一类（之前禁用缓存的）救回来一半。

这三件事任何一件单独做完，理论上命中率都能再涨 5-10 个百分点；三个全做完，50% 是有希望的。但每一件的工程复杂度都比这一篇讲的 4 个坑高一个量级，**不是所有团队都值得做**。

## 九、4 周落地路线

如果你看到这里决定动手，下面是按周排好的落地路线，每周给一个验收指标。

### 第 1 周：埋点 + 当前命中率基线测量

**目标**：把"命中率"这个数字测准。如果连基线都没有，后面所有改动都是凭感觉。

**做的事**：

- 在网关层加 4 个埋点：`request_in / cache_lookup_hit / cache_lookup_miss / llm_call_done`
- 把这些埋点接到 Prometheus / ClickHouse
- 拉出一周的数据，按租户 / 按时间分布画命中率曲线
- 跑一份"误命中率审计"：人工抽 200 条命中样本判断答案是否对得上 query

**验收指标**：

- 命中率曲线在监控面板上 24/7 可见
- 知道当前命中率的均值、P50、P95、P99（典型值大概率在 8-15% 区间）
- 误命中率 < 5%（如果 > 5% 立刻降低阈值或停用缓存，这是事故等级）

### 第 2 周：Embedding 阈值调优（坑 1 闭环）

**目标**：把命中率拉到 20%+，误命中率压到 1% 以下。

**做的事**：

- 准备 2000-5000 条人工标注的"等价 / 不等价" pair（如果有 LLM 自动标注的预算可以扩到 1 万条）
- 用第 3 章的 ROC 脚本跑一次，确定阈值
- 跑模型对比：bge-large-zh-v1.5 vs 你当前用的，决定是否切模型
- 如果数据量足够，可以微调一版（可选）

**验收指标**：

- 命中率 ≥ 20%
- 误命中率 ≤ 1%
- ROC 图存档（Q3 复盘要用）

### 第 3 周：双层 Key 改造（坑 2）

**目标**：把命中率拉到 28%+，租户隔离合规事故 = 0。

**做的事**：

- 设计 L1 Key（tenant_id + model + prompt_v + param_hash）和 L2 Key（embedding）
- 改造缓存读写接口，所有调用方升级
- 上线"通用知识共享分区"开关（先不开，灰度再开）
- 写一份租户隔离测试用例：模拟 A 租户问 B 租户的私有问题，确认完全隔离

**验收指标**：

- 命中率 ≥ 28%
- 跨租户测试用例 100% 通过
- 旧缓存失效后命中率回升时间 < 7 天

### 第 4 周：分级 TTL + singleflight（坑 3+4）

**目标**：把命中率拉到 35%+，命中率波动 ≤ ±3pp。

**做的事**：

- 整理"内容时效矩阵"，产品 + 研发 + 客服三方共建
- 上线分级 TTL 路由器，REALTIME 类直接禁用缓存
- 上线 singleflight + TTL 抖动
- 跑一次预热脚本（top 5000 query）

**验收指标**：

- 命中率 ≥ 35%
- "实时数据回错"投诉 = 0
- LLM 厂商 429 错误数下降 80%+
- 命中率峰谷波动 ≤ ±3pp

到这里就把这一篇文章里的 4 个坑全部闭环了。再之后是第 8 章里那三件"50%+ 路线"的事——那些不在 4 周内能做完，等命中率稳在 38% 一两个月后再启动。

## 十、写在最后

回到开场 OpenRouter 那张图：**44.6 万亿 Token、连续 8 周上涨**。这条曲线短期内不会停——大模型已经从"工程师炫技"变成了"业务系统的水电"，调用量只会继续涨。

行业涨了多少不重要，重要的是**你的账单**怎么涨。这一篇的核心论点其实就一句话：

> **路由解决"用什么模型"，缓存解决"还需不需要调"——前者节省单价，后者节省调用次数，两者乘起来是 10 倍量级。**

上一篇我把路由表展开来讲，从 12% 到 38% 这一篇把缓存展开来讲。如果你的团队这两件事都做完了，单位 Token 成本能从行业均值降到 1/10 不到——这不是夸张，是我们半年实实在在跑出来的数字。

我特别想强调的一点是：**这 4 个坑里没有一个是"模型能力不够"导致的**。阈值是工程问题、Key 是工程问题、TTL 是工程问题、击穿是工程问题。LLM 厂商再卷、模型再便宜，都帮你解决不了这些坑——这是中后端工程师能拿在手里的、唯一能完全决定的成本控制权。

最后一句话留给打算动手的同行：**4 周路线不长，但它真的会让你下个季度的账单变好看**。我们走过这条路，留了一份配套源码在 GitHub，里面有这一篇文章里所有代码的可运行版本，包括 ROC 校准脚本、双层 Key 缓存、分级 TTL 路由器、singleflight 实现、预热脚本。建议你 fork 一份照着改，不要从零写。
