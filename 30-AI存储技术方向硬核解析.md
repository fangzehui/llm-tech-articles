# AI 存储技术方向硬核解析：HBM4、CXL、QLC 122TB、存算一体、KV-Cache 卸载、NVMe-oF 六大主线全景（2026-07 定稿版）

上一篇《大模型存储市场 2024-2026》讲了钱、讲了盘、讲了 HBM 涨价三年翻四倍这个"果"。这一篇讲"因"——AI 存储这一年里，到底跑到了哪一步、还差哪几层皮。别整虚的，六条主线：HBM3E/HBM4、CXL 3.x 内存池化、QLC 122TB 大盘、存算一体（PIM/CIM）、KV-Cache 四级卸载、RDMA / NVMe-oF 训练网络。每一条都掐了近半年的一手信息，配套三段可跑代码放在 GitHub。

---

## 一、为什么 AI 让"内存墙 / 存储墙"重新成为主角

CPU 圈子讲了三十年"存储墙"，一直没成为主要矛盾——通用计算靠 cache 层级和 out-of-order 执行糊得过去。AI 训推把这层遮羞布一把扯了。

一颗 H100 SXM 的 BF16 峰值算力 989 TFLOPS，HBM3 带宽 3.35 TB/s，Roofline 平衡点 arithmetic intensity ≈ 295 FLOPs / Byte。B200 FP8 4500 TFLOPS 配 8 TB/s HBM3E，平衡点抬到 563；GB300 FP4 15 PFLOPS 配 8 TB/s，平衡点飙到 1875。这个数字意味着什么？意味着 LLM decode 阶段每读 1 byte KV 只做 1 次乘加，实际 arithmetic intensity ≈ 1，比平衡点低两到三个数量级——GPU 有 99% 的时间在等内存搬运。这不是我说的，是 NVIDIA Dynamo 团队在博文里承认的："GPU memory is limited and costly. As the prompt length increases, the KV Cache grows larger... Holding large KV Caches in GPU memory for long durations is not scalable"（据 NVIDIA Developer Blog《[How to Reduce KV Cache Bottlenecks with NVIDIA Dynamo](https://developer.nvidia.com/blog/how-to-reduce-kv-cache-bottlenecks-with-nvidia-dynamo)》）。

再看训练侧。Meta 在 SIGCOMM 2024 那篇 [《RDMA over Ethernet for Distributed AI Training at Meta Scale》](https://cs.stanford.edu/~keithw/sigcomm2024/sigcomm24-final246-acmpaginated.pdf) 里给了答案：单集群 24K+ GPU、400G RoCE、专有后端网络，光是 200G 升 400G 就要重新调 DCQCN 参数好几轮，"performance was degraded with default DCQCN settings and doubled ECN thresholds"。带宽升上去，拥塞控制没跟上，收敛速度反而掉。

结论——AI 时代的存储不是可选项。它是决定 GPU 利用率、TCO、乃至能不能训得动大模型的第一变量。下面六条主线，就是业界这两年扑上去解决这个第一变量的六个方向。

## 二、HBM 主线：HBM3E → HBM4，2026 年"元年"里发生了什么

**HBM3E 12H 已成 2025-2026 主力。** 三星官网列的规格：36 GB / 12H 堆叠 / up to 9.2 Gbps / 单栈最大 1180 GB/s（据 [Samsung Semiconductor HBM 页面](https://semiconductor.samsung.com/kr/dram/hbm/)）。美光的 12H HBM3E 用 1β 工艺，功耗比上代降低约 12%，已通过 NVIDIA B300 认证，2025 Q3 起大批量交付 AWS Trainium 3 和谷歌 Ironwood TPU（据雪球[《高带宽存储 HBM》](https://xueqiu.com/8291133521/355850684)整理的公开信息）。

**HBM4 是 2026 的关键跳跃点。** I/O 引脚数从 1024 直接翻倍到 2048。三星在 [2026 年 3 月 HBM4 信息图](https://news.samsungsemiconductor.com/kr/영상-인포그래픽으로-보는-hbm4/) 中给出的官方数据：11.7 Gbps 稳定运行、最大 13 Gbps，单栈带宽 3.3 TB/s，比 HBM3E 提升约 2.7 倍；同一秒钟能传 750 部 4 GB 的电影。三星用 1c DRAM（10 nm 级第 6 代）做核心 die，用自家 4 nm 逻辑工艺做 base die——这是 IDM 模式的甜头，SK 海力士得靠台积电 N4/N3 代工。

**量产节奏。** 据 [TechNews 报道](https://technews.tw/2025/12/08/sk-hynix-has-adjusted-both-the-timeline-and-production-capacity-for-hbm4-mass-production-scheduled-for-2026/)，SK 海力士 HBM4 原计划 2026 年 2 月量产、Q2 末扩产，因 NVIDIA Rubin 平台节奏调整而后延至 3-4 月量产、Q3 扩产；2026 上半年 HBM3E 仍是 SK 海力士主力。三星则反向抢跑，2026 年 2 月已首家出货 HBM4；市场传 SK 海力士将拿走 NVIDIA HBM4 订单的 60%~70%（据 [studio-hx 博客](https://blog.studio-hx.io/posts/hbm4-mass-production-rubin-round)）。美光 12H HBM3E 已通过 B300 认证，HBM4 目标 2026 年下半年跟进。

**16H HBM4 已经在门口。** SK 海力士在 [CES 2026 展台](https://news.skhynix.com/sk-hynix-showcases-next-generation-ai-memory-innovations-at-ces-2026/) 首度公开 16 层堆叠、48 GB 的 HBM4 样品。三星[5 月底交付首批 12 层 HBM4E 样品](https://news.sina.cn/bignews/insight/2026-07-01/detail-iniffytk3182973.d.html)，带宽做到 4.0 TB/s、容量 48 GB。

**散热是下一个卡脖子点。** SK 海力士 5 月 26 日发布 [iHBM 技术](https://news.skhynix.com.cn/ihbm-solution/)，把冷却元件（ICE）直接嵌入 HBM 封装，D2D PHY 区域热阻降低 30% 以上。这条线未来会决定 HBM5 能不能继续往上堆。

**GPU 平台侧对应：** H200 SXM 141 GB HBM3E / 4.8 TB/s；B200 192 GB HBM3E / 8 TB/s；GB300 NVL72 单 GPU 8 TB/s；AMD MI350X 288 GB HBM3E / 8 TB/s；NVIDIA Rubin R100 目标 26 TB/s（HBM4 3.3 TB/s × 8 stack）。这张表构成了未来 12-18 个月所有 AI Infra 采购的参考坐标。

## 三、CXL 3.x 内存池化：KV-Cache 的"第三层电梯"

HBM 再猛，容量摆在那里——B200 才 192 GB，一个 671B MoE 模型加 128K 上下文和 batch=32 的 KV cache 就顶天。所以要有第二级、第三级。

CPU DRAM 是第二级：DDR5-6400 单通道 51.2 GB/s，单 socket 12 通道能到约 614 GB/s，带宽还行，但容量受 DIMM 槽和内存控制器数量限制，一般单机 2-4 TB 到顶。

**CXL 3.x 内存池化是第三级新的解。** CXL 3.1 支持交换机拓扑、多主机共享同一内存池、内存分级迁移，延迟通常 200-500 ns（近似 NUMA 远端），带宽由 PCIe 6.0 决定，x16 单向 64 GB/s。这个数字远低于 HBM 但远高于 SSD，正好在 KV-Cache 分层中卡到"够大够快"的甜蜜点。

**阿里云 2025 年底论文 Beluga 是目前最硬的公开实证。** 论文 [《Beluga: A CXL-Based Memory Architecture for Scalable and Efficient LLM KVCache Management》](https://arxiv.org/html/2511.20172v2) 提出：GPU 与 CPU 通过 CXL 交换机共享大规模内存池，用 CXL fabric 的 load/store 语义直接访问，延迟接近本地内存，同时避免 RDMA 通信协议开销。Beluga-KVCache 相比现有 CPU DRAM 方案实现约 89% 的吞吐提升（论文摘要节选）。这直接证明——CXL 池不是 PPT，是量产系统。

**耶鲁 + 哥大 FPGA '26 的 CXL-SpecKV 是另一条路。** 论文 [《CXL-SpecKV: A Disaggregated FPGA Speculative KV-Cache for Datacenter LLM Serving》](https://dl.acm.org/doi/pdf/10.1145/3748173.3779188) 把 KV-Cache 卸到远端 FPGA 内存，配套推测执行和压缩，比 GPU-only 基线吞吐提升 3.2 倍、内存成本降 2.8 倍。

CXL 3.1 已是 2025 年正式发布的标准，2026 年主流 CPU 平台（Xeon 6 / EPYC 9005+）都原生支持；CXL Consortium 网站在推 3.2 规范。就 KV-Cache 场景而言，CXL 已经从"未来"变成"下一代 AI 服务器的默认配置"。

## 四、QLC 122TB 大盘：AI 数据湖的"底座换血"

训练数据从 TB 级涨到 PB 级、EB 级，机械盘（HDD）已经拖后腿——QLC 大容量 SSD 是接盘侠。

**Solidigm D5-P5336 122.88 TB PCIe 4.0** 是目前商用 SSD 天花板（据 [Solidigm 官网 D5-P5336 页面](https://www.solidigmtech.com.cn/products/data-center/d5/p5336.html)）。规格：122.88 TB、4K 随机读最高 1005K IOPS、TCO 优于 HDD 阵列约 36%。产品线覆盖 U.2、E1.L、E3.S 三种 form factor。厂家还给了一个诡异的耐久指标——按 122 TB / 18K IOPS 4K 随机写连续跑 5 年，SSD 仍有剩余寿命。

**Kioxia LC9 系列 245.76 TB E3.L** 把上限直接抬到 SSD 容量的两倍。据 [Kioxia 2026-05-15 新闻稿](https://www.kioxia.com/ja-jp/business/news/2026/20260515-1.html)，Kioxia 与戴尔联合发布，Dell PowerEdge R7725xd 2U 服务器装 40 个 LC9 245.76 TB SSD，单机达成 **9.8 PB / 2U**——这个密度往前推 5 年不敢想。

**MinIO × Solidigm ExaPOD 参考架构** 给了一个可复用的对象存储蓝图。据 [Solidigm 白皮书《The New Economics of AI Factory Efficiency》](https://www.solidigm.com/content/dam/solidigm/en/site/products/technology/2026/new-economics-of-ai-factory-minio/document/new-economics-of-ai-factory-graph.pdf)：ExaPOD 可扩展到 1 EB 可用容量 / 640 台服务器 / 32 个机柜，聚合吞吐达 19.2 TB/s，首字节延迟中位数 3-5 ms，尾延迟保持在中位数的 ~11% 以内。8 客户端并发时最坏 1 秒桶下的吞吐依然在健康区间——QLC 大盘 + S3 协议已经是 AI 训练数据湖的默认答案。

**PLC（5-bit / cell）在路上**，但 2026-2027 仍以 QLC 为主。行业更值得关注的是 HBF（High Bandwidth Flash）——SK 海力士与闪迪 [2026-02-26 联合宣布](https://sk.com.cn/5-1SKnew-main.html?id=1499062897400086528) 在 OCP 框架下成立标准化联盟，HBF 定位介于 HBM 和 SSD 之间的新存储层级，工程样片 2027 年面世，NVIDIA、Google 最快 2028 年采用。这不是"更快的 SSD"，是"NAND 直接堆到 HBM 高度、当持久内存用"的架构。

## 五、存算一体（PIM / CIM）：把运算搬到存储里

前四条都在解决"怎么让数据更快搬到 GPU"。存算一体反过来问："能不能不搬？"

**Samsung HBM-PIM** 是最早公开的商业化尝试，2021 年就在 HBM 颗粒里嵌入 AI 处理单元；2026 年在 [Samsung Semiconductor 官网](https://semiconductor.samsung.com/kr/dram/hbm/) 上仍作为演进方向。

**SK 海力士 GDDR6-AiM 和 AiMX 加速卡** 已到样品阶段。据 [SK Hynix CES 2026 新闻稿](https://news.skhynix.com/sk-hynix-showcases-next-generation-ai-memory-innovations-at-ces-2026/)，AiMX 是基于 GDDR6-AiM 芯片的加速卡原型，专门针对 LLM 场景优化。同一新闻还提到 CuD（Compute-using-DRAM）——在 cell 内做简单计算加速数据处理；以及 CMM-Ax（CXL Memory Module-Accelerator xPU）——把计算能力集成到 CXL 内存中，是 CXL + PIM 的一次合流。

**PIM 的关键难点：编程模型和生态。** 论文 [《Amplifying Effective CXL Memory Bandwidth for LLM Inference via Transparent Near-Data Processing》](https://arxiv.org/pdf/2509.03377v1) 提出的 CXL-NDP 方案，用 plane-aware KV cache 布局在 CXL 内存内做透明压缩和精度切换，不需要修改 CXL.mem 协议。这种"host 不感知的近数据处理"是 PIM 走向主流的关键姿势——如果每次跑 PIM 都要改上层框架，那这个技术永远进不了生产。

**PIM 短期看不会成为主流数据中心 GPU 的替代**，但会以 SoC IP、加速卡形态渗透进推理侧。韩国政府 [2026-06-29 宣布](https://news.sina.cn/bignews/insight/2026-07-01/detail-iniffytk3182973.d.html) 的 800 万亿韩元投资中，专门有 30 万亿韩元 15 年的存算一体研发预算——这是国家层面的下注。

## 六、KV-Cache 卸载：本地 HBM → CPU DRAM → CXL 池 → SSD 四级电梯

前面几节讲硬件，这一节讲把这些硬件缝起来的软件层——KV-Cache 分层卸载。这也是 GitHub 配套源码的第一段代码。

**LMCache 是目前最完整的开源分层缓存。** 据其 [官方架构文档](https://docs.lmcache.ai/developer_guide/architecture.html)，LMCache 实现了 GPU / CPU DRAM / 本地 disk / 远程存储四级层次：GPU 保工作集，CPU DRAM 作 hot cache（pinned memory 加速传输），本地磁盘或 NVMe GDS 缓大对象，Redis / Mooncake / InfiniStore 做持久层。技术报告[《LMCache: An Efficient KV Cache Layer for Enterprise-Scale LLM Inference》](https://lmcache.ai/tech_report.pdf) 里两个细节很硬：一是 vLLM 原生的 KV cache page 是 20-63 KB 的小单元，不足以打满带宽，LMCache 用 chunk-level（默认 256 tokens）批传输解决；二是 KV cache 传输容易与推理并发竞用 CUDA stream，LMCache 用独立 stream + 定制 CUDA kernel 拆开。

**NVIDIA Dynamo + NIXL 走另一条路。** 据 [Dynamo 博文](https://developer.nvidia.com/blog/how-to-reduce-kv-cache-bottlenecks-with-nvidia-dynamo)：Dynamo 通过 NIXL 低延迟传输库，把 KV cache 从 GPU 直接搬到 CPU RAM、本地 SSD 或远程网络存储，不打断推理。这是 vLLM `OffloadingConnector` 之外的商业级路径。

**llm-d 项目把这一切标准化。** [llm-d 官方文档](https://llm-d.ai/docs/architecture/advanced/kv-management/kv-offloader) 把 KV-Cache offloading 抽象成两个能力："capacity"（HBM 只有几十 GB，CPU RAM 加一个数量级，存储近乎无限扩展）和 "sharing"（本地缓存节点隔离，共享存储让跨节点缓存复用、新副本快速拉起）。文档给了一个引人的数字：加载已缓存 block 通常比重新计算快 **最多 16 倍**。

**分层的经济学。** CSDN 上 [《KV-Cache 技术详解：大模型推理性能提升 24 倍的完整指南》](https://blog.csdn.net/m0_59614665/article/details/154533520) 引用的实测数据：LMCache 在 10K tokens 中等长度上下文场景下，TTFT 显著下降，吞吐最高 24 倍提升；GPUDirect Storage（GDS）比 non-GDS 吞吐高 32.5%-96.6%；带宽提升与系统吞吐几乎线性正相关，多数场景带 9.1%-28.6% 增益。

配套源码第 1 段 `kv_cache_tiered_offload.py` 就实现了这套模型。用它跑 671B MoE + 128K 上下文 + batch=32 + kv=8.5 KB/token 的场景，KV 总量约 34 GB，全落在 L1 GPU HBM 内；改到 batch=256、上下文 200K、kv=10 KB/token，KV 总量涨到约 488 GB，L1 装 192 GB、L2 CPU DRAM 装剩下的 296 GB，重加载时延约 480 ms 量级——不快但可用，比 recompute 128K 上下文快至少一个量级。

## 七、RDMA / NVMe-oF：训练集群的存储血管

有了 HBM、CXL 池、QLC 大盘、KV cache 分层，还要有把它们连起来的"血管"——存储网络。AI 训练场景对存储网络的要求，和传统 HPC 完全不是一回事。

**Meta 的 400G RoCE 部署** 是目前公开最完整的实战。据 SIGCOMM 2024 论文，Meta 用 RoCE v2 支撑 24K GPU 大集群、O(10K) 训练规模、多个集群；后端网络 rail-optimized，200G 升 400G 过程中 DCQCN 默认参数导致性能下降，需要放宽 ECN 阈值、逐轮微调。这个 lesson 很典型：**AI 存储网络不是"更快的以太网"，是需要单独调优的一个物种**。

**NVIDIA DGX SuperPOD B300 XDR 参考架构** 是 vendor 侧标杆。据 [NVIDIA 官方文档](https://docs.nvidia.com/dgx-superpod/reference-architecture/scalable-infrastructure-b300-xdr/latest/storage-architecture.html)：DL 训练的关键 I/O 是 re-read；模型规模越大，checkpoint 写入越关键，"checkpoint 文件动辄 TB 级，虽不频繁写，但通常同步写、阻塞训练前进"。这决定了训练存储网络必须同时优化"高并发读"（数据加载）和"突发写吞吐"（checkpoint）两个方向。

**DDN AI400X2** 是这套架构在存储侧的经典搭配。据 [Lenovo 官方参考架构文档](https://lenovopress.lenovo.com/lp2021.pdf)：AI400X2 单机 NVMe 配置 30-500 TB，多机聚合到数百 PB 单命名空间，用 QM9790 InfiniBand 400G NDR fabric，ConnectX-7 400 Gb/s 网卡，rail-optimized fat-tree 拓扑，非阻塞设计。

**中国 AI 集群侧的对齐。** 极客天成 400G IB 并行存储 [公开文档](https://blog.csdn.net/Wulalalaaaaa/article/details/145120566) 给出实测：单节点带宽从 200 Gb/s 提升到 400 Gb/s，端到端延迟 <600 ns，用 RDMA 零拷贝绕过 CPU。国内头部厂商（阿里、腾讯、字节）都在从 200G 升级到 400G IB / RoCE，字节的 MegaScale、腾讯的星脉、阿里的 HPN 都有类似技术栈。

**800G 是下一个坎。** Stanford 的[《RDMA over Ethernet for Distributed Training at Meta Scale》教学材料](https://yangzhou1997.github.io/teaching/ecs289d_fall25_lecture5_meta_rdma.pdf)提到：ML workload 正在往 800G NIC、102.4 Tbps 交换机、Co-Packaged Optics（CPO）方向卷。NVIDIA Quantum-X800（800G XDR）已经在 2025-2026 年落地。

配套源码第 3 段 `checkpoint_nvmeof_estimator.py` 把这套算清楚：DeepSeek-V3 671B BF16 + Adam 优化器 = 6248 GB 单次全局 checkpoint；用 400G IB 单口（50 GB/s）落盘（0.7 有效系数）需 178 秒；升到 800G IB 单口需 89 秒；聚合到单节点 8 端口 800G（800 GB/s 目标）只需 11 秒。以每天 checkpoint I/O 不超过训练总时间 5% 为约束，671B 用 400G 单口一天最多 checkpoint 24 次，用 8 端口 800G 能到 380+ 次。带宽升级直接决定训练 fault tolerance 的经济性。

## 八、选型矩阵：小 / 中 / 大三档训推场景配比

理论讲完，落到桌面上。三档参考配置（截至 2026-07-06 的主流货架产品）：

**小档（7B-70B 推理、单卡到单机 8 卡）：**
- GPU：H200 SXM 或 B200 单机 8 卡
- HBM：不用改，用官方默认
- CPU DRAM：1-2 TB DDR5-6400
- CXL：可选，一般不上
- SSD：4-8 块 U.2 15 TB TLC（做 KV cache L3）
- 网络：200G IB 或 100G RoCE 即可
- 适用：内部工具、小规模 API 服务、Agent 后端

**中档（70B-400B 训练 / 千亿级 MoE 推理）：**
- GPU：H200 / B200 集群 64-256 卡
- HBM：全 HBM3E
- CPU DRAM：单节点 3-4 TB
- CXL：**必上**，单节点扩 4-8 TB CXL 内存池，配 LMCache / Beluga 类方案做 KV L3
- SSD：Solidigm 61.44 TB 或 122.88 TB QLC 大盘，每节点 8-16 块，做 KV L4 和数据湖
- 网络：400G IB NDR，rail-optimized fat-tree
- 适用：企业级私有 LLM 推理、中等规模训练

**大档（千亿-万亿 MoE 训练、超长上下文推理）：**
- GPU：GB300 NVL72 或 B300 XDR，1K-10K 卡起
- HBM：全 HBM3E，2026 H2 后混部 HBM4
- CPU DRAM：单节点 8-16 TB
- CXL：CXL 3.1 交换机构建跨节点内存池，容量 100 TB 级
- SSD：Kioxia LC9 245.76 TB E3.L 或 Solidigm ExaPOD 级 QLC 阵列
- HBF（前瞻）：2027-2028 引入
- 网络：800G IB XDR，Quantum-X800 交换机；跨 rack 用 CPO
- 适用：GPT-5 级前沿模型训练、Anthropic / OpenAI / DeepSeek 级 lab

一句话总结：**HBM 决定推理延迟，CXL 决定 KV 上限，QLC 决定训练数据体量，NVMe-oF 决定容错节奏，PIM 决定下下代能耗。** 缺一环，都会成为整个 AI 集群的短板。

## 九、六个常见误区，别踩

写到这里，把工程实践里最容易被拍脑袋误判的六个坑单独拎出来。这一节是我最想在会议上按住某些采购负责人肩膀讲的。

**误区一：HBM 容量涨了就等于 KV cache 装得下。** 从 HBM3E 24 GB 到 HBM4 36 GB、再到 48 GB 的 16H HBM4，单栈容量确实翻了一番。但 KV cache 增长速度更猛——上下文从 8K 涨到 128K、再到 1M，KV 线性膨胀，200 万 token 上下文的单请求 KV 可以到几十 GB。指望"下一代 HBM 大点就够了"是错觉，KV 必须走分层卸载。这也是为什么 vLLM、SGLang、TensorRT-LLM 全都在 2025-2026 加了 KV offload connector 接口，而不是干等 HBM 变大。

**误区二：CXL 只是"更慢的 DRAM"。** 很多人把 CXL 类比成 NUMA 远端节点，认为"多花钱、更慢，没用"。但 CXL 3.x 的核心价值不在带宽，而在**池化和多主机共享**。同一份 KV cache 在 CPU DRAM 里必须每台机器复制一份；在 CXL 池里可以多个 GPU node 共享一份、连"KV cache 复用（cross-request cache hit）"的命中率都能提升。阿里云 Beluga 那 89% 提升，一半功劳在此。CXL 是架构价值，不是纯带宽价值。

**误区三：QLC 耐久性不够，训练不能用。** 早年 QLC 的 DWPD（每天全盘写入次数）低到 0.1-0.2 让人担忧。但 AI 数据湖 workload 是**读密集、写稀疏**——数据集写入一次、读几百上千个 epoch。Solidigm 官方按 122 TB / 18K IOPS 4K 随机写连续跑 5 年后 SSD 仍有余量的耐久指标，对 AI workload 足够到过剩。反倒是 checkpoint 这种"周期性大写入"场景，得用另配的 TLC 缓冲盘，别混用。

**误区四：所有训练网络都得上 400G / 800G。** 中小规模（≤128 GPU）训练用 200G HDR 完全够用，Meta 论文里也提过 200G 阶段稳定跑了很久。400G / 800G 的价值在 O(1K) 以上超大集群、跨 rack 通信占比高的场景。盲目上 400G 反而要重调 DCQCN 参数、多花预算，得不偿失。选型第一步是先跑 nccl-tests 摸清 all-reduce / all-to-all 到底在多少 GPU 之后开始瓶颈，再决定网络代际。

**误区五：存算一体（PIM）就在拐点上。** 不是。PIM 有真实价值，但商业化拐点还没到——原因不是硬件不成熟，是**软件栈 / 编译器 / 算子库**没跟上。CUDA 生态是 NVIDIA 十几年砸出来的，PIM 目前每一款产品的编程接口都不同、算子覆盖率低、部分场景需要重写。2026-2027 是"技术验证"，2028+ 才有可能进主流推理集群。别提前 all-in。

**误区六：SSD 大就够了，不用管协议栈。** 122 TB QLC 盘扔进服务器不等于"AI 数据湖就绪"。真正的性能取决于三层：**文件系统层**（GPFS / Lustre / WekaFS / Vast Data / MinIO 的元数据能力）、**协议层**（S3 / NFS / POSIX 的语义匹配）、**GPUDirect Storage 支持**（能不能绕过 CPU 直接落 GPU HBM）。同样的物理盘，配 Vast Data 或 Weka 能榨出 3-5 倍的实际训练吞吐。硬件是必要条件，不是充分条件。

## 十、总结与观察点

写到最后，AI 存储这一年发生的事情，本质上是**一次跨了三十年的旧规范的翻新**：DRAM 标准（HBM 从 JEDEC HBM3 到 HBM4）、互连标准（PCIe 5→6、CXL 2→3.1→3.2）、NAND 密度（TLC 32 TB → QLC 122 TB → 245 TB）、网络标准（IB 200G→400G→800G，以太网 RoCE 齐头并进）、计算模型（存算分离 → 存算一体）——五个方向同时挪动。这在 IT 史上并不常见，上一次同等规模的翻新大概是 2005-2010 年 SSD / 万兆以太网 / 虚拟化那波。

未来 12 个月三个观察点：

- **2026 H2：** NVIDIA Rubin 平台首批上市、HBM4 12H 大批出货，能不能撑住 5000 tokens/s/GPU 的推理目标是核心指标。
- **2027 H1：** CXL 3.2 正式定稿，多层次内存池化商业方案铺开；同时 HBF 工程样片首露面，看是不是能撑住 SK 海力士画的饼。
- **2027 H2 至 2028：** 存算一体（PIM / CMM-Ax）能否在推理侧拿到第一个真正的量产订单，比如从 Meta / Google / 阿里的推理集群里冒头。

那句老话：**别只盯着 GPU，盯着 GPU 边上的存储和网络。**大多数模型跑不起来，都是被这两条腿绊倒的。
