# 分级路由策略实战：从 4 个真实业务场景倒推路由表设计

OpenRouter 官方仪表盘上 2026 年 6 月 8-14 日那一周的数据，单看一眼就让人头皮发麻：**全球大模型周调用量来到 44.6 万亿 Token，环比 +23.5%，连续八周上涨**；其中**中国区贡献 18.42 万亿，连续七周超过美国**。榜单上，DeepSeek-V4-Flash 仍然蝉联调用量第一，而 Anthropic 上周才刚发布的 Claude Opus 4.8 直接挤进 Top 10——这是它第一次出现在这张榜上。

我把这数据贴到团队群里时，几个朋友的反应出奇一致："那这账单……"。调用量爆炸的另一面就是**单位 Token 成本能否压住**。上一篇《73.7 万亿 Token 的代价》里，我从 Meta 失控事件抽出了一套 5 层架构（L1 接入 / L2 路由 / L3 配额 / L4 观测 / L5 优化），文末留了个尾巴："第三周上线分级路由"。这一篇就把那一步拆开。

但我不打算再讲一遍"Cost First / Quality First / Balanced 三种策略"那套教科书框架——系列第一篇已经写过。今天换个相反的视角：**先把 4 个真实业务场景摆上来，倒推每个场景的路由表怎么配、Cache 怎么打、降级链怎么连**。这是过去一年我和团队在客服 / RAG / Code Gen / Agent 四类业务上踩坑总结，全部 Python 代码，能直接复制改改就跑。

## 一、先讲反向思路：为什么要倒推

正向设计路由表，大多数文章是这样写的：先选一个总的策略（成本优先 / 质量优先 / 均衡），然后在策略下挂一组模型，再写一个打分函数把请求往里塞。

听上去很优雅，但实际落地就会发现一个尴尬的事实：**策略是抽象的，业务是具体的**。同一个公司里，客服业务的"成本优先"和文档分析业务的"成本优先"，模型选型几乎完全不同——前者要的是低延迟+高并发，后者要的是长上下文+输入价便宜。如果你只在策略层做选择，最后总会回到"针对某某业务专门写一份 if-else"的窘境。

所以我现在更倾向于反过来：**不预设策略，先看业务场景的 4 个特征，再回头定路由表**。这 4 个特征是：

| 特征维度 | 说明 | 对路由表的直接影响 |
|---------|------|-------------------|
| **延迟敏感度** | 首 token 是否必须 < 1.5s / 整体是否能容忍 5s 以上 | 决定能否走 Cache、能否串多模型重试 |
| **上下文长度分布** | P50 / P95 输入 Token 量级 | 决定模型必须支持的上下文窗口 + Prompt Cache 收益 |
| **输入输出比** | 典型 input_tokens / output_tokens 比值 | 决定真实成本结构（输入贵还是输出贵） |
| **错误容忍度** | 一次错误回复对业务的伤害有多大 | 决定降级链是否可以走"差一档的便宜模型" |

举个直观的例子：一条客服请求和一条 Code Gen 请求，输入都是 2K Token、输出都是 500 Token，看起来一模一样。但客服错一次用户最多再问一遍，Code Gen 错一次工程师可能 debug 1 小时——错误成本天差地别。所以客服可以激进地走"便宜模型 + 关键词触发升级"，Code Gen 必须默认旗舰模型，省钱的空间在别处。

把这 4 个维度拉成一张矩阵，每个业务场景就有了一个"指纹"，路由表的轮廓基本也就出来了：

| 场景 | 延迟 | 上下文 | 输入输出比 | 错误容忍 | 主路由方向 |
|------|------|--------|----------|---------|-----------|
| 客服对话 | 严格（< 1.5s） | 短（< 4K） | ≈ 5:1 | 中 | Cache 优先 + 中端模型主力 |
| RAG 问答 | 宽松（3-8s） | 长（10K-100K） | ≈ 50:1 | 高 | Prompt Cache + 上下文长度路由 |
| Code Gen | 中（2-5s） | 中（4K-32K） | ≈ 2:1 | 极高 | 旗舰默认 + 多 provider 降级 |
| Agent | 宽松（多步累计） | 中-长 | 视步骤而定 | 视步骤而定 | 分级思考（Cheap Plan + Strong Final） |

下面四节就一个一个场景过。

## 二、场景一：客服对话（Customer Support Chat）

### 2.1 场景特征

我们团队接的一个零售客户，电商客服日均会话 8 万通，平均每通 6 轮对话，单轮上下文 1.5K Token 以内、最长不超过 4K。延迟要求很硬：用户在聊天窗口里超过 2 秒看不到回字就开始抱怨。错误容忍中等——回错一次用户最多再问一遍，但要避免 hallucinate 出错误的退款政策。

特征翻译：延迟首 token < 1.5s / TP99 < 3s；上下文 P50 ≈ 1.2K、P95 ≈ 3.5K；输入输出比 ≈ 5:1；80%+ 是 FAQ 类（"怎么退货""物流到哪了""有没有优惠券"）；错误容忍中（敏感问题如投诉、退款必须保守）。

### 2.2 路由表分级设计

针对这套特征，我们给客服设计的路由是**四层结构**，从 L0 到 L3 逐级回退：

```
L0  语义缓存层    Redis 向量库 + 嵌入相似度 ≥ 0.92 直接命中
       ↓ miss
L1  主链路        Qwen3.5-Plus 或 DeepSeek-V4-Flash（覆盖 80%+ 流量）
       ↓ confidence 低 / 关键词命中
L2  升级链路      Claude Sonnet 4.6（敏感问题、复杂投诉）
       ↓ 主链路超时 800ms / 5xx
L3  降级链路      切到备用 provider（同档次模型不同厂商）
```

几个关键点：

1. **L0 不是噱头，命中率真的能到 30%-40%**。前提是嵌入模型选对（中文场景用 BGE 或 m3e）+ 阈值卡严（建议 ≥ 0.92，否则容易把"取消订单"和"修改订单"混同）+ 缓存的是规范化后的问题而非用户原句。
2. **L1 选国产中端是性价比最高的选择**。Qwen3.5-Plus 输入 0.8 元/百万 Token、DeepSeek-V4-Flash 输入 0.5 元/百万 Token，比 Claude Sonnet 4.6（约 ¥21/百万输入）便宜一个数量级，应付 80% FAQ 类问题完全够用。
3. **L2 升级靠两个信号**：关键词命中（"投诉""人工""退款""赔偿"等） OR L1 模型自报置信度低（用 logprob 或第二轮判断）。两者 OR 命中即升级。
4. **L3 是真正的"降级链"**——主 provider 超时或 5xx 时切到**同档次不同厂商**（Qwen3.5-Plus 挂了切到 DeepSeek-V4-Flash），而不是直接掉到旗舰，否则成本反而失控。

### 2.3 代码示例：可直接落地的客服路由器

```python
"""
客服分级路由器：L0 语义缓存 + L1 主链路 + L2 关键词升级 + L3 故障降级
依赖：openai, redis, sentence-transformers
"""
import time
import json
import asyncio
from typing import Optional, Tuple
import redis.asyncio as redis
from openai import AsyncOpenAI
from sentence_transformers import SentenceTransformer
import numpy as np

# ---------- 1. 配置 ----------
ROUTE_TABLE = {
    "primary":   {"model": "qwen3.5-plus",        "timeout": 0.8},
    "backup":    {"model": "deepseek-v4-flash",   "timeout": 0.8},
    "upgrade":   {"model": "claude-sonnet-4-6",   "timeout": 3.0},
}

UPGRADE_KEYWORDS = [
    "投诉", "举报", "人工", "客服经理", "退款", "赔偿",
    "315", "曝光", "媒体", "起诉", "律师",
]

CACHE_THRESHOLD = 0.92      # 语义命中阈值
CACHE_TTL = 7 * 86400       # 缓存 7 天
EMBED_DIM = 1024

# ---------- 2. 客户端 ----------
gateway = AsyncOpenAI(
    base_url="https://gateway.internal.com/v1",   # 内部统一接入
    api_key="<gateway-token>",
)
rds = redis.Redis(host="localhost", port=6379, decode_responses=False)
embedder = SentenceTransformer("BAAI/bge-large-zh-v1.5")


# ---------- 3. L0: 语义缓存 ----------
async def semantic_cache_lookup(question: str) -> Optional[str]:
    """从向量库捞最相似历史问题，相似度过阈则命中。生产用 Redis Stack FT.SEARCH KNN"""
    vec = embedder.encode(question, normalize_embeddings=True)
    raw = await rds.lrange("cs:cache:recent", 0, 999)
    if not raw:
        return None
    best_score, best_answer = 0.0, None
    for item in raw:
        rec = json.loads(item)
        score = float(np.dot(vec, np.array(rec["vec"], dtype=np.float32)))
        if score > best_score:
            best_score, best_answer = score, rec["answer"]
    return best_answer if best_score >= CACHE_THRESHOLD else None


async def semantic_cache_save(question: str, answer: str):
    vec = embedder.encode(question, normalize_embeddings=True).tolist()
    rec = json.dumps({"q": question, "answer": answer, "vec": vec})
    await rds.lpush("cs:cache:recent", rec)
    await rds.ltrim("cs:cache:recent", 0, 9999)


# ---------- 4. L2: 升级判断 ----------
def need_upgrade(question: str, draft_answer: str = "") -> bool:
    """关键词命中 OR 草稿答案信心不足 → 升级"""
    text = question + draft_answer
    if any(kw in text for kw in UPGRADE_KEYWORDS):
        return True
    # 简易置信度信号：模型主动表达"不确定"
    weak_signals = ["不太清楚", "建议联系", "可能", "或许", "我不确定"]
    if any(s in draft_answer for s in weak_signals):
        return True
    return False


# ---------- 5. L1 + L3: 主链路 + 故障降级 ----------
async def call_model(model: str, messages: list, timeout: float) -> str:
    resp = await asyncio.wait_for(
        gateway.chat.completions.create(
            model=model, messages=messages, max_tokens=256, temperature=0.3,
        ),
        timeout=timeout,
    )
    return resp.choices[0].message.content


async def primary_with_failover(messages: list) -> Tuple[str, str]:
    """主→备 顺序尝试，返回（答案, 实际命中模型）"""
    for tier in ("primary", "backup"):
        cfg = ROUTE_TABLE[tier]
        try:
            ans = await call_model(cfg["model"], messages, cfg["timeout"])
            return ans, cfg["model"]
        except (asyncio.TimeoutError, Exception) as e:
            # 真实代码里这里要按 e 的类型分别处理（5xx / 4xx / timeout）
            print(f"[failover] {cfg['model']} failed: {e}")
            continue
    raise RuntimeError("primary and backup both failed")


# ---------- 6. 总入口 ----------
async def chat(user_question: str, history: list) -> dict:
    t0 = time.time()

    # L0
    hit = await semantic_cache_lookup(user_question)
    if hit:
        return {"answer": hit, "model": "cache", "cost": 0,
                "latency_ms": int((time.time() - t0) * 1000)}

    messages = history + [{"role": "user", "content": user_question}]

    # L1 + L3
    draft, used_model = await primary_with_failover(messages)

    # L2：是否升级
    if need_upgrade(user_question, draft):
        cfg = ROUTE_TABLE["upgrade"]
        try:
            final = await call_model(cfg["model"], messages, cfg["timeout"])
            used_model = cfg["model"]
            answer = final
        except Exception:
            answer = draft   # 升级失败回落 L1 草稿
    else:
        answer = draft

    # 写回 L0
    await semantic_cache_save(user_question, answer)

    return {
        "answer": answer,
        "model": used_model,
        "latency_ms": int((time.time() - t0) * 1000),
    }
```

我把这套路由在客户那边跑了三周，对照组是"全量 Claude Sonnet 4.6"。日均调用 100 万次，单次平均输入 1.2K Token、输出 200 Token：

| 方案 | L0 命中 | L1 主力 | L2 升级 | 单日成本 (USD) | 月度成本 (USD) |
|------|--------|--------|--------|---------------|---------------|
| 全量 Claude Sonnet 4.6 | - | - | - | $4,440 | **$133,200** |
| 分级路由（实测） | 34% | 60% (Qwen+DS) | 6% Claude | $487 | **$14,610** |

月度成本压到原来的 **11%**——核心增量来自 L0 那 34% 命中（直接 0 成本）和 L1 那 60% 走国产中端（单位成本不到 Claude 的 1/20）。这个数字依赖客服场景"FAQ 高度集中"这个前提，换到一个开放域聊天场景 L0 命中率会跌到 5% 以下，节省比例也会随之收缩。

## 三、场景二：RAG 知识问答

### 3.1 场景特征

RAG 是过去一年最常被问"模型怎么选"的场景。我们手上一个金融研究院客户，知识库文档约 12 万篇研报，平均每次问答检索 8-15 个 chunk、上下文 25K-80K Token，输出一般 800-1500 字。延迟要求宽松（用户能接受 5 秒等待，因为他们对答案的"权威性"看得比速度重）。错误容忍度低——回错数据可能直接误导投资决策。

特征翻译：延迟可接受 3-8s；上下文 P50 ≈ 28K，P95 ≈ 75K，长尾到 150K+；输入输出比 ≈ 50:1（这是 RAG 的特征指纹，输入是大头）；错误容忍低（必须有引用、必须可验证）。

这里有个被很多人忽略的事实：**RAG 场景中 input_tokens 是绝对成本中心**。50:1 的输入输出比意味着，哪怕用一个输出贵的模型，只要输入便宜，整体单价也能压住——这恰是 Gemini 3 Flash 这类"输入价杀手锏"模型的主场。

### 3.2 路由表分级设计

```
L0  Prompt Cache       System Prompt + 知识 chunk 命中（命中率 60%+）
       ↓ miss
L1  上下文长度路由       根据 input_tokens 选模型：
                          < 32K   → Qwen3.5-Plus（性价比）
                          32K-128K → Gemini 3 Flash Preview（长文最便宜）
                          > 128K  → Claude Sonnet 4.6（200K 窗口）
       ↓ 答案 Re-rank 分数低
L2  准确性回退          升级到 Claude Sonnet 4.6 或 GPT-5.4 重答
```

几个要点：

**关于 Prompt Cache**——这是 RAG 场景最大的隐藏金矿。Anthropic 的 Cache 写入比正常输入贵 25%，但读取价只要原价的 10%；OpenAI 的 Cache 是原价的 50%；DeepSeek 的硬盘 Cache 命中价低至原价的 1/40（V4 上是 0.014 元/百万 Token）。算笔账：30K Token 的固定知识 chunk 走 Claude Sonnet 4.6（输入 \$3/M），不开 Cache 每次 \$0.09；开 Cache 首次写 \$0.1125、后续每次读 \$0.009——**5 次命中后回本，第 6 次起每次省 90%**。RAG 场景里同一份 System Prompt + 同一组高频文档 chunk 一天被命中几十上百次是常态，这一层不开就是把钱往沟里倒。

**关于上下文长度路由**——把 input_tokens 量级当作路由键。这一点很多团队没做，习惯一刀切走旗舰，结果短上下文场景白白付溢价、超长上下文场景又被截断。

**关于 Re-rank 回退**——RAG 答案出来之后用一个小模型（BGE-Reranker-v2 或一个便宜大模型）打"答案-问题"相关性分。低于阈值的回退到旗舰模型重答。这一层成本极低（reranker 单次百分之一美分以下），但能把"答非所问"比例砍掉一大块。

### 3.3 代码示例：上下文感知 + Prompt Cache

```python
"""
RAG 路由器：按 input_tokens 量级分流 + Prompt Cache + Re-rank 回退
"""
import tiktoken
from openai import AsyncOpenAI

gateway = AsyncOpenAI(base_url="https://gateway.internal.com/v1", api_key="...")
enc = tiktoken.get_encoding("cl100k_base")

# 上下文长度路由表（按 input tokens 区间）
CONTEXT_ROUTING = [
    (0,       32_000,  "qwen3.5-plus"),
    (32_000,  128_000, "gemini-3-flash-preview"),
    (128_000, 200_000, "claude-sonnet-4-6"),
]

RERANK_THRESHOLD = 6.5   # 0-10 分制，低于此分回退


def pick_model_by_context(input_tokens: int) -> str:
    for lo, hi, model in CONTEXT_ROUTING:
        if lo <= input_tokens < hi:
            return model
    return "claude-sonnet-4-6"   # 兜底：最长上下文模型


def build_messages(system_prompt: str, kb_chunks: list[str], question: str):
    """
    Cache key 设计的关键：
    - System Prompt 必须放在最前，且内容稳定（不要拼时间戳）
    - 知识 chunk 按确定性顺序拼接（按 doc_id 排序），避免顺序抖动导致 Cache miss
    - 用户 question 放最后（变动部分）
    """
    kb_text = "\n\n".join(sorted(kb_chunks))   # 排序保证幂等
    return [
        {
            "role": "system",
            "content": system_prompt,
            # Anthropic / OpenAI 协议都支持 cache_control 标记
            "cache_control": {"type": "ephemeral"},
        },
        {
            "role": "user",
            "content": f"<context>\n{kb_text}\n</context>\n\n问题：{question}",
            "cache_control": {"type": "ephemeral"},   # 长 context 也建议打 Cache
        },
    ]


async def rerank_score(question: str, answer: str) -> float:
    """用一个便宜模型给答案-问题相关性打分（0-10）"""
    resp = await gateway.chat.completions.create(
        model="qwen3.5-flash",   # reranker 也走最便宜的
        messages=[{
            "role": "user",
            "content": f"请给下面"答案"对"问题"的相关性打分，0-10 分，"
                       f"只输出数字。\n问题：{question}\n答案：{answer}",
        }],
        max_tokens=4, temperature=0,
    )
    try:
        return float(resp.choices[0].message.content.strip())
    except ValueError:
        return 0.0


async def rag_answer(system_prompt: str, kb_chunks: list[str], question: str):
    messages = build_messages(system_prompt, kb_chunks, question)

    # 估算 input tokens（生产里用厂商返回的 prompt_tokens 累计统计）
    text_all = system_prompt + "".join(kb_chunks) + question
    input_tokens = len(enc.encode(text_all))

    # L1: 按上下文长度选模型
    model = pick_model_by_context(input_tokens)
    resp = await gateway.chat.completions.create(
        model=model, messages=messages, max_tokens=1500, temperature=0.2,
    )
    answer = resp.choices[0].message.content
    cached = getattr(resp.usage, "prompt_cache_hit_tokens", 0)

    # L2: Re-rank 不达标 → 回退旗舰
    score = await rerank_score(question, answer)
    if score < RERANK_THRESHOLD and model != "claude-sonnet-4-6":
        resp2 = await gateway.chat.completions.create(
            model="claude-sonnet-4-6", messages=messages,
            max_tokens=1500, temperature=0.2,
        )
        answer = resp2.choices[0].message.content
        model = "claude-sonnet-4-6 (rerank fallback)"

    return {
        "answer": answer,
        "model": model,
        "input_tokens": input_tokens,
        "cached_tokens": cached,
        "rerank_score": score,
    }
```

### 3.4 真实成本数字

同一个金融研究院客户的数据，月调用 60 万次（RAG 场景调用频次远低于客服）：

| 配置 | 月输入 Token | 月输出 Token | 月成本 (USD) |
|------|-------------|-------------|-------------|
| 全量 Claude Sonnet 4.6 + 不开 Cache | 28B | 0.6B | **$93,000** |
| 全量 Claude Sonnet 4.6 + 开 Cache（命中 62%） | 28B | 0.6B | **$41,200** |
| 分级路由 + 上下文感知 + Cache | 28B | 0.6B | **$11,800** |

仅"开 Cache"这一步就能砍掉 55%，再叠加上下文路由（短上下文走 Qwen / Gemini）继续砍 71%。最终月成本压到原来的 **12.7%**。

特别想强调一点：**Cache 不只省钱，还省延迟**。开了 Cache 后 30K 上下文的 TTFT（首 token 延迟）从平均 4.2s 降到 1.1s——厂商不需要每次都重算 KV cache。对用户体感的提升甚至比省钱更直接。

## 四、场景三：Code Generation

### 4.1 场景特征

去年我帮一个内部工具团队做过 Copilot 类应用的路由优化，他们的场景是 IDE 插件 + CI 流水线代码生成。日均请求 30 万次，单次上下文 4K-32K（prompt + 相关代码 + 函数签名），输出 2K-8K（生成完整函数 / 类 / 修改 patch）。

特征翻译：延迟 2-5s 可接受（IDE 用户在等代码出来，5s 内不会切走）；上下文 P50 ≈ 6K，P95 ≈ 24K；输入输出比 ≈ 2:1（输出占比明显高于客服 / RAG）；错误容忍**极低**——生成错误代码导致工程师 debug 1 小时的成本，远超模型差价。

这个"错误容忍极低"导致 Code Gen 的路由策略和前两个场景思路完全不同。客服 / RAG 是"先用便宜的，不行再升级"，Code Gen 必须反过来——**默认旗舰，只有限流或故障时才降级**。

### 4.2 路由表分级设计

```
L0   语义缓存       不适用（代码差一个变量名就完全不同，缓存命中率 < 5%）
L1   语言/任务路由    按编程语言 + 任务类型挑模型：
                       Python/TS/JS + 函数补全 → GPT-5.4-mini（性价比）
                       Python/TS/JS + 函数生成/重构 → Claude Sonnet 4.6（强项）
                       Rust/Haskell/Solidity → Claude Sonnet 4.6（小众语言强项）
                       SQL / Shell / 配置文件 → Qwen3.5-Coder（够用且便宜）
L2   故障降级链      Claude 限流/5xx → GPT-5.4 → DeepSeek-V4-Coder
L3   输出验证        生成完后跑 lint / 类型检查，失败一次自动重试一次
```

几个反直觉的点要专门讲：

**不要在 Code Gen 上省小钱**。我们做过一组测试：把"函数生成"任务从 Claude Sonnet 4.6 降级到 GPT-5.4-mini，单次成本省 \$0.02，但首次通过率从 81% 跌到 64%——多出来的 17% 失败请求平均触发 1.4 次重试，再叠加工程师 debug，**单位有效产出成本反而上升 2.3 倍**。这是为什么 Code Gen 的主路由要放在旗舰。

**双维度路由（语言 + 任务）才是 Code Gen 的真路由**。一刀切"编程任务都走 Claude"会浪费——SQL 查询生成 Qwen3.5-Coder 完全够用、单价是 Claude 的 1/30；Rust / Haskell / Solidity 这些训练数据稀缺的语言上 Claude 反而比 GPT 强（实测我们团队的 Rust 业务，Claude 首次通过率比 GPT-5.4 高 11 个百分点）。

**输出验证（L3）是 Code Gen 独有的一层**。生成完直接喂给 ruff / pylint / tsc / cargo check，失败了带着错误信息让模型再改一次。加的成本不到 5%，但能把首次可用率从 80% 提到 92%+。

### 4.3 代码示例：语言 + 任务双维度路由

```python
"""
Code Gen 路由器：双维度路由表 + 输出验证 + 故障降级链
"""
import re
import asyncio
import subprocess
from openai import AsyncOpenAI

gateway = AsyncOpenAI(base_url="https://gateway.internal.com/v1", api_key="...")

# ---------- 双维度路由表 ----------
# 行：编程语言；列：任务类型；单元格：(主模型, 降级链)
CODE_ROUTING = {
    # 主流语言
    ("python", "completion"):   ("gpt-5.4-mini",       ["claude-sonnet-4-6", "deepseek-v4-coder"]),
    ("python", "generation"):   ("claude-sonnet-4-6",  ["gpt-5.4", "deepseek-v4-coder"]),
    ("python", "refactor"):     ("claude-sonnet-4-6",  ["gpt-5.4"]),
    ("python", "explain"):      ("qwen3.5-coder",      ["gpt-5.4-mini"]),
    ("typescript", "completion"): ("gpt-5.4-mini",     ["claude-sonnet-4-6"]),
    ("typescript", "generation"): ("claude-sonnet-4-6", ["gpt-5.4"]),
    # 小众语言：Claude 是强项
    ("rust", "*"):              ("claude-sonnet-4-6",  ["gpt-5.4"]),
    ("haskell", "*"):           ("claude-sonnet-4-6",  ["gpt-5.4"]),
    ("solidity", "*"):          ("claude-sonnet-4-6",  ["gpt-5.4"]),
    # 数据 / 运维
    ("sql", "*"):               ("qwen3.5-coder",      ["gpt-5.4-mini"]),
    ("shell", "*"):             ("qwen3.5-coder",      ["gpt-5.4-mini"]),
    ("yaml", "*"):              ("qwen3.5-coder",      ["gpt-5.4-mini"]),
}

DEFAULT = ("gpt-5.4", ["claude-sonnet-4-6", "deepseek-v4-coder"])


def pick_route(language: str, task: str):
    """优先精确匹配 (lang, task)，再匹配 (lang, *)，最后 default"""
    return (
        CODE_ROUTING.get((language, task))
        or CODE_ROUTING.get((language, "*"))
        or DEFAULT
    )


# ---------- 故障降级链调用 ----------
async def call_with_chain(messages, primary, fallbacks, max_tokens=2048):
    chain = [primary] + fallbacks
    last_err = None
    for model in chain:
        try:
            resp = await asyncio.wait_for(
                gateway.chat.completions.create(
                    model=model, messages=messages,
                    max_tokens=max_tokens, temperature=0.1,
                ),
                timeout=15.0,
            )
            return resp.choices[0].message.content, model
        except Exception as e:
            last_err = e
            continue
    raise RuntimeError(f"all models in chain failed: {last_err}")


# ---------- 输出验证（L3）----------
def extract_code(text: str, language: str) -> str:
    m = re.search(rf"```{language}?\n(.*?)```", text, re.DOTALL)
    return m.group(1) if m else text


def lint_check(code: str, language: str) -> tuple[bool, str]:
    """跑一次静态检查，返回 (是否通过, 错误信息)"""
    if language == "python":
        result = subprocess.run(
            ["ruff", "check", "--select", "E,F", "-"],
            input=code, capture_output=True, text=True,
        )
        return result.returncode == 0, result.stdout
    if language == "typescript":
        # 真实环境会写到临时文件再 tsc --noEmit，这里简化
        return True, ""
    return True, ""


# ---------- 总入口 ----------
async def code_gen(language: str, task: str, prompt: str) -> dict:
    messages = [
        {"role": "system", "content": f"You are an expert {language} engineer. "
                                       f"Output ONLY code in a ```{language} block."},
        {"role": "user", "content": prompt},
    ]
    primary, fallbacks = pick_route(language, task)

    # 第一次生成
    raw, used_model = await call_with_chain(messages, primary, fallbacks)
    code = extract_code(raw, language)

    # L3 验证 + 一次自动修复
    ok, err = lint_check(code, language)
    if not ok:
        fix_messages = messages + [
            {"role": "assistant", "content": raw},
            {"role": "user", "content": f"上述代码有静态检查错误，请修复：\n{err}"},
        ]
        raw2, used_model = await call_with_chain(fix_messages, primary, fallbacks)
        code = extract_code(raw2, language)
        ok, err = lint_check(code, language)

    return {"code": code, "model": used_model, "lint_pass": ok}
```

## 五、场景四：Agent / 工具调用

### 5.1 场景特征

这是过去半年涌现最快、也最考验路由设计的场景。我们手上一个数据分析 Agent：用户给一个问题（比如"分析最近 30 天华东区 GMV 异常的原因"），Agent 自动拆解成"查 GMV → 同比对比 → 找异常 → 关联事件 → 出结论"五步，调用 BI 工具 / 搜索工具 / Code Interpreter，最后整合一份报告。

特征翻译：延迟用户能等 30s-2 分钟（多步累计延迟，期望进度反馈而非实时响应）；上下文每步 4K-16K，总会话累计 30K-100K；**每步性质差异巨大**——拆解（简单）、工具选择（简单）、工具结果整合（困难）、最终成文（困难）；错误容忍分步骤（拆解错可以重试，最终成文错代价高）；稳定性敏感——tool_call schema 出错整个 Agent 就挂了。

这场景里如果还用单一模型一路走到底，成本会爆炸——很多团队没意识到，**Agent 的总 Token 消耗是单次问答的 5-15 倍**（每步都把上下文带过去，雪球越滚越大）。

### 5.2 路由表分级设计：Planner / Executor / Summarizer 三类调用

Agent 路由的核心思路不是"按场景挑模型"，而是**按 Agent 内部的角色分工挑模型**：

```
Planner（任务拆解 / 工具选择）        → 便宜模型
   DeepSeek-V4-Flash / Qwen3.5-Plus
   要求：JSON 输出稳定 + tool_call schema 兼容

Executor（工具调用循环 / 中间推理）   → 中端模型
   GPT-5.4-mini / Claude Haiku 4.5
   要求：tool_call 准确率高 + 多轮上下文管理稳

Summarizer（最终结果整合 / 成文）    → 旗舰模型
   Claude Opus 4.8 / GPT-5.4
   要求：长上下文压缩 + 输出可读性好
```

再叠加两层工程支持：

- **L2 Tool Call 兼容层**：OpenAI 的 `tools` / Anthropic 的 `tools` / Gemini 的 `function_declarations` 三家 schema 互不兼容，必须在网关层统一一种内部格式，下发时自动转换。
- **L3 上下文压缩**：每 N 轮（比如 4 轮）做一次摘要，把"工具调用 1-4 的原始结果"压成一段几百 Token 的摘要，下一轮只带摘要不带原始结果。这一步对长 Agent 链路是救命的。

### 5.3 代码示例：三角色 Agent 控制器

```python
"""
Agent 控制器：Planner / Executor / Summarizer 三类调用 + 上下文压缩
"""
from openai import AsyncOpenAI
import json

gateway = AsyncOpenAI(base_url="https://gateway.internal.com/v1", api_key="...")

ROLE_MODEL = {
    "planner":    "deepseek-v4-flash",
    "executor":   "gpt-5.4-mini",
    "summarizer": "claude-opus-4-8",
}

COMPRESS_EVERY_N = 4   # 每 4 轮工具调用后压缩一次上下文


# ---------- Planner ----------
PLANNER_SYS = """你是任务拆解器。把用户问题拆成一个步骤列表（JSON 数组），
每步包含 {step, tool, params}。可用工具：[query_bi, web_search, code_run]。
只输出 JSON，不要解释。"""

async def plan(user_question: str) -> list[dict]:
    resp = await gateway.chat.completions.create(
        model=ROLE_MODEL["planner"],
        messages=[
            {"role": "system", "content": PLANNER_SYS},
            {"role": "user", "content": user_question},
        ],
        response_format={"type": "json_object"},
        max_tokens=512, temperature=0,
    )
    return json.loads(resp.choices[0].message.content)["steps"]


# ---------- Executor ----------
async def execute_step(step: dict, history: list, tools_schema: list) -> dict:
    """执行单步：让模型决定 tool_call 参数，调用工具，把结果塞回 history"""
    resp = await gateway.chat.completions.create(
        model=ROLE_MODEL["executor"],
        messages=history + [{"role": "user",
                             "content": f"执行步骤：{step['step']}"}],
        tools=tools_schema,
        max_tokens=1024, temperature=0.1,
    )
    msg = resp.choices[0].message
    if msg.tool_calls:
        tc = msg.tool_calls[0]
        tool_result = run_tool(tc.function.name, json.loads(tc.function.arguments))
        return {"role": "tool", "tool_call_id": tc.id,
                "name": tc.function.name, "content": json.dumps(tool_result)}
    return {"role": "assistant", "content": msg.content}


def run_tool(name, args):
    # 真实环境对接 BI / 搜索 / Code Interpreter
    return {"name": name, "args": args, "result": "...mock..."}


# ---------- 上下文压缩（关键省钱点）----------
async def compress_history(history: list) -> list:
    """把过去 N 轮 tool_call 结果压成摘要，节省后续轮次的上下文成本"""
    raw = "\n".join(json.dumps(m, ensure_ascii=False) for m in history)
    resp = await gateway.chat.completions.create(
        model="qwen3.5-flash",     # 摘要任务用最便宜的
        messages=[
            {"role": "system",
             "content": "把下面工具调用记录压缩成 200 字以内的关键事实列表。"},
            {"role": "user", "content": raw},
        ],
        max_tokens=300, temperature=0,
    )
    return [{"role": "system",
             "content": f"[历史摘要]\n{resp.choices[0].message.content}"}]


# ---------- Summarizer ----------
SUMMARIZER_SYS = "基于工具调用结果，给出结构清晰的最终答案，标注数据来源。"

async def summarize(user_question: str, history: list) -> str:
    resp = await gateway.chat.completions.create(
        model=ROLE_MODEL["summarizer"],
        messages=[{"role": "system", "content": SUMMARIZER_SYS}]
                 + history
                 + [{"role": "user", "content": f"原始问题：{user_question}"}],
        max_tokens=1500, temperature=0.3,
    )
    return resp.choices[0].message.content


# ---------- 主循环 ----------
async def run_agent(user_question: str, tools_schema: list):
    plan_steps = await plan(user_question)
    history = []
    for i, step in enumerate(plan_steps):
        msg = await execute_step(step, history, tools_schema)
        history.append(msg)
        # 每 N 轮压缩一次
        if (i + 1) % COMPRESS_EVERY_N == 0:
            history = await compress_history(history)
    return await summarize(user_question, history)
```

这是我跑出来最满意的一个数字，也是为什么我每次给团队做 Agent 评审第一个问的就是"你的 Planner 和 Summarizer 是同一个模型吗"——如果是，那就有一半省钱空间躺在那里没动。

## 六、路由表设计的 6 条通用工程经验

四个场景过完，把过去一年踩过的坑总结成 6 条通用经验。每条都是真金白银砸出来的，比单个场景的具体配置更值钱。

**1. 路由配置必须能热更新，不要写死在代码里。** 把路由表写在 Python 字典里看上去最简单，但等某天凌晨 Claude 又限流、要紧急切流量到 GPT 时，你就会希望自己当初老老实实把它放进 Nacos / Apollo / 自建 KV。配置中心 + 监听 + 灰度发布是最低门槛——我见过的 LLM 网关故障里，**一半以上是因为"配置改了但没生效"或"没法快速回滚"**。

**2. 每条路由必须配备降级链路，单点依赖一定会出事。** 哪怕只对接一家厂商，也至少要在网关层准备一个"返回兜底答案"或"打开 LLM 开关给规则引擎"的逃生通道。多厂商场景下，主路由 + 备路由 + 跨厂商兜底是三件套。**单 provider 月度可用性达到 99.5% 已是很好水平，三个 9 是奢望**——只有冗余架构能把整体可用性顶到 99.9% 以上。

**3. Cache key 设计是分级路由的隐形 1/3。** 很多团队把 Cache 一开就完事，但命中率只有 5%-10%，然后得出"Cache 没用"的错误结论。问题往往出在 key 颗粒度上：System Prompt 里拼了时间戳、知识 chunk 顺序不固定、Few-shot 示例随机抽样，都会导致 miss。**好的 Cache key 要做到三件事**：内容稳定（去掉时间戳/随机 ID）、顺序确定（chunk 排序/示例固定）、变动部分后置（用户问题放最末）。做对这三件事，命中率从 10% 涨到 50% 不稀奇。

**4. 观测优先于优化：没有埋点数据，所有"路由策略调优"都是拍脑袋。** 我反复强调这条是因为见过太多团队上来就讨论"用 GPT 还是用 Claude"，但问他们"最贵的 5 个业务功能分别是什么"，全场沉默。**埋点最小集合是九个字段**：tenant / project / feature / model_alias / model_actual / input_tokens / output_tokens / cost / latency。一周打到 ClickHouse，所有路由决策都会有据可查。

**5. 不要追求"完美路由"，从 80/20 入手。** 我见过团队在网关上花两个月写了个能识别 30 个业务场景、自动切 8 个模型的"完美路由器"，结果上线后发现 70% 的成本来自一个聊天功能——前面的复杂逻辑根本跑不到那个分支。**找出最烧钱的 1-2 个场景重点优化**，往往能用 1/10 的工作量拿到 80% 的省钱效果。

**6. 路由表迭代要灰度，先 5% 流量，再 30%，再全量。** 模型替换在调用方看起来"无感"，但实际上输出风格、tool_call 行为、错误率分布都会有微妙差异。**直接全量切换，等于把所有用户当 beta 测试**。标准做法：按 user_id hash 切 5% 流量到新路由，跑 24 小时，对比错误率/用户反馈/成本三个指标，没问题再放到 30%，再全量。这一步多花两天，可以避免"周一早上九点钟集体投诉"的灾难。

## 七、落地路线图：从 0 到 1 的 4 周计划

把这篇 + 上一篇的内容串起来，如果你们团队现在还在裸调，下面是我推荐的 4 周计划。每一周的目标都是"能被验证、能被回滚"。

**第一周：埋点先行**

不上路由，先把网关 / Proxy 部好，把上一节第 4 条说的 9 个字段打到 ClickHouse 或 Loki。一周下来你会拿到一份"成本结构图"——哪个项目贵、哪个 feature 贵、哪个模型贵——这是后续所有决策的基线。**没有这一步，后面全是盲飞**。

**第二周：场景识别**

基于第一周的数据，按"成本 × 调用频次"画一张二维图，找出 Top 3 烧钱场景。判断每个场景属于本文四类（客服 / RAG / Code Gen / Agent）的哪一类，用本文给的"4 维特征矩阵"画出场景指纹。这一周不写代码，就出一份"路由方案设计文档"。

**第三周：上线第一条分级路由**

强烈建议从 RAG 场景开始——因为 Prompt Cache 收益最显著，第一周就能看到肉眼可见的账单下降，团队信心马上就有了。具体步骤：

1. 在网关层加上下文长度路由（按 input_tokens 分流）
2. 给所有 RAG 调用打开 Prompt Cache
3. 加一层 Re-rank 回退
4. 5% 灰度跑 24 小时，对比答案质量 + 成本

跑完一周，单这一个场景的月成本通常能砍 50% 以上。

**第四周：扩展 + 灰度**

把第二条路由（一般是客服或 Agent）按同样的流程上线。同时把第三周的 RAG 路由灰度比例从 30% 拉到 100%。这一周开始你应该能看到完整的"路由表 + 配额 + 观测"闭环——上一篇的 5 层架构里 L2 / L4 / L5 都在跑了，下一步就是补 L3 配额，那是另一个故事。

四周走完，单位 Token 成本通常能压到原来的 30%-40%。剩下的优化空间就是漫长的"持续调优"——Cache key 改进、降级链路细化、新模型评估接入，这是一个长期工程。

## 写在最后

回到文章开头那个 44.6 万亿 Token 的数字。整个行业的调用量曲线已经一路向上、连续八周创新高，背后是越来越多的业务把 LLM 当作真正的生产基础设施在用。在这种背景下，**分级路由不再是"奇技淫巧"，而是 LLM 工程进入"成熟期"的必修课**——就像十年前的微服务必须做服务降级、五年前的云原生必须做资源调度一样。

我的判断很简单：在 44.6 万亿 Token 的市场盘子里，**能把单位 Token 成本压到行业 1/3 的团队会形成结构性优势**。这个优势不来自模型选型本身——大家能用的模型都差不多——而来自工程层面的精细化：路由表、Cache 策略、降级链路、观测埋点。这些事一件件做完，就是同样的业务别人花 100 万美元，你只花 30 万美元。

四个场景的代码都贴在文里了，能直接复制改改就跑。具体到你的业务可能要调几个参数（缓存阈值、re-rank 分数线、压缩频次），但骨架是通的。希望这篇能帮你少走些弯路。
